## Лаба 1 — Каталог книг (Модели, Шаблоны, Теги)

Здесь я сделал мини-сайт каталога книг на Django. Одна модель — `Book` с полями: название, автор, жанр, цена, описание, год издания и ссылка на обложку.

Данные лежат в SQLite, а на странице книги рендерятся сеткой в три колонки. Если у книги нет обложки — рисуется CSS-заглушка с градиентом и названием поверх.

Шаблоны используют наследование: есть базовый `base.html`, от него расширяется `book_list.html` через `{% extends %}`.

**Встроенные теги/фильтры** которые я применил: `upper`, `capfirst`, `default` и `truncatewords` — последний обрезает длинное описание до 20 слов.

**Кастомные (свои)** — три штуки, лежат в `catalog_tags.py`:
- `format_price` — фильтр, форматирует цену в формат «1 200,00 ₽» с пробелами-разделителями.
- `is_new_book` — фильтр, вызывает доменный метод `is_new_release()` и возвращает `True`, если книга издана в последние 3 года. В шаблоне по этому флагу рендерится бейдж «Новинка».
- `get_reading_time` — простой тег, считает слова в описании и делит на 180 (средняя скорость чтения), выводит «X мин. чтения».

## Лаба 2 — Корзина, Покупатели, Админка

Добавил три новые сущности: **Покупатель** (`Buyer`), **Товар** (это те же книги из первой лабы) и **Корзина** (`Cart`). Корзина связана с покупателем через `ForeignKey`, а с книгами через `ManyToManyField`.

В **админке** зарегистрировал все модели. У корзины настроил:
- `list_display` — выводит ID, покупателя, количество товаров, статус и дату.
- Количество товаров — вычисляемое поле через метод `get_books_count()`, оно считает `obj.books.count()`.
- `search_fields` — поиск по имени и почте покупателя через `buyer__name`, `buyer__email` (двойное подчёркивание — это Django делает JOIN по связанной таблице).
- `list_filter` — фильтрация по статусу корзины и дате.

На фронте сделал страницу `/carts/` — таблица со всеми корзинами, информацией о покупателе, списком книг и итоговой суммой. Сумму считает доменный метод `calculate_total_cost()`, а отображает её кастомный фильтр `format_price` из первой лабы.

## Лаба 3 — Промежуточная модель, Инлайны, Вычисляемые поля

Главная идея — теперь у каждой книги в корзине есть **количество**. Для этого я заменил обычный `ManyToManyField` на связь через **промежуточную модель** `DjangoCartItem` (параметр `through`). У неё три поля: `cart`, `book` и `quantity`.

Зачем это нужно: обычный M2M создаёт скрытую таблицу только с двумя ID. Чтобы хранить дополнительные данные (количество), нужно объявить таблицу явно.

В **админке** добавил `TabularInline` для `DjangoCartItem` — теперь прямо в карточке корзины можно добавлять, удалять книги и менять количество. Подключается через `inlines = [DjangoCartItemInline]`.

**Вычисляемые поля** в списке корзин:
- `get_total_quantity` — суммарное количество единиц товара.
- `get_total_cost` — итоговая стоимость (цена × количество для каждой позиции, потом сумма).

Оба считаются через доменные методы, не в SQL.

Ещё добавил `date_hierarchy = "created_at"` — это навигационная панель по датам вверху списка в админке. Неактивные корзины визуально приглушаются на сайте через CSS-класс `.cart-inactive` (полупрозрачность), а в админке статусы выделяются цветными бейджами через `format_html`.

Для оптимизации запросов использую `prefetch_related("items__book")` — это решает проблему N+1 запросов. Без него Django сделал бы по отдельному запросу на товары каждой корзины, а с ним — всего два запроса на всё.

## Лаба 4 — Портал заявок и Middleware

Полностью новое приложение — **корпоративный портал заявок** (`tickets`). Сотрудники создают заявки на внутренние услуги (IT-поддержка, ремонт, бухгалтерия), прикладывают файлы, оставляют комментарии, меняют статус.

**Модели** (их шесть):
- `Ticket` — заявка (название, описание, сервис, статус, вложение).
- `Comment` — комментарий к заявке.
- `TicketHistory` — автоматическая история изменений полей (что изменилось, старое и новое значение).
- `UserProfile` — профиль с ролью (user / moderator / admin).
- `UserSettings` — персональные настройки (тема, уведомления).
- `AuditLog` — журнал аудита действий.

**Три роли:** обычный пользователь, модератор, администратор. Права зависят от роли: автор может редактировать свою заявку только в статусе «Новая» и только 2 часа после создания. Модератор и админ могут менять статусы.

Помимо HTML-страниц, есть **REST API** — два эндпоинта (`TicketApiListView`, `TicketApiDetailView`), которые отдают данные в JSON через `JsonResponse`.

Основная тема этой лабы — **пять Middleware** (промежуточных обработчиков):

1. **RequestIdMiddleware** — генерирует UUID для каждого запроса и прокидывает его в заголовок ответа `X-Request-ID`. Нужен для отладки.
2. **AuditLogMiddleware** — пишет в БД кто зашёл, с какого IP, каким браузером, на какую страницу, каким методом и какой статус вернулся.
3. **RateLimitingMiddleware** — защита от спама. Считает запросы по IP через кэш Django. Если больше 15 POST/API запросов за минуту — отдаёт 429.
4. **AutoUserMiddleware** — автологин для удобства тестирования. Если не авторизован — логинит как `demo_user`.
5. **BusinessHoursMiddleware** — запрещает опасные операции (POST, PUT, DELETE) вне рабочего времени (до 9:00, после 18:00, в выходные). Администраторов не трогает.

Middleware работают как конвейер: запрос проходит сверху вниз по списку в `settings.py`, ответ — снизу вверх.
