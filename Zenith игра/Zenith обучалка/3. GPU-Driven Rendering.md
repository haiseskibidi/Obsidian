# Глава 3. GPU-Driven Rendering & Оптимизация VRAM

В прошлый раз мы выяснили, что постоянные телефонные звонки процессора на OpenGL-завод (**Draw Calls**) и дерганья рабочих с перенастройкой станков (**State Changes**) убивают производительность игры напрочь.

Теперь разберём вершину графической инженерии движка **Zenith** — технологию **GPU-Driven Rendering (Рендеринг под управлением видеокарты)**.

Мы сделаем так, чтобы директор (CPU) вообще перестал звонить рабочим на конвейер на каждый чих. Вместо этого мы научим видеокарту саму читать команды отрисовки прямо из видеопамяти и рисовать весь гигантский воксельный мир за один-единственный системный вызов.

---

## 1. Парадигма GPU-Driven: Полная автоматизация

Вспомним нашего Умного Директора (CPU) и Автоматический Завод (GPU).

В классическом подходе (CPU-Driven) директор стоит над душой у рабочих:
* *«Так, рабочие, возьмите VBO (Vertex Buffer Object — буфер вершин, хранящий координаты и параметры геометрии) чанка №1... Нарисуйте его!»* (glDrawElements)
* *«Теперь бросьте его, возьмите VBO чанка №2... Нарисуйте его!»* (glDrawElements)

В **GPU-Driven** подходе мы внедряем полную автоматизацию:
1. **Глобальный пул мешей (Mesh Pooling):** Мы сваливаем все детали (геометрию чанков) на один огромный склад в видеопамяти (VRAM).
2. **Непрямая отрисовка (Indirect Drawing):** Директор больше не звонит лично. Он берет бумагу и пишет один раз толстую книгу-план на весь день: *«Нарисовать меш с полки №1, потом меш с полки №2, потом меш с полки №3...»*. Эта книга отправляется на склад видеокарты (**Indirect Buffer**).
3. **Групповой вызов (MultiDraw batching):** Директор делает ровно один звонок на завод и говорит: *«Ребята, начните работу и строго следуйте книге-плану, которую я вам скинул!»* (**MultiDraw Indirect**).

Рабочие завода открывают книгу и параллельно собирают миллионы объектов сами. Директор свободен, а видеокарта загружена на все 100%.

## 2. Архитектура Mesh Pooling: Единый склад VRAM

В обычных играх для каждого воксельного чанка создаётся свой маленький буфер VBO и IBO (Index Buffer Object — буфер индексов, задающий порядок соединения вершин). Если в зоне видимости 1000 чанков, видеокарте приходится делать тысячи переключений буферов в секунду. Это приводит к серьёзной фрагментации видеопамяти и тормозам шины PCIe (Peripheral Component Interconnect Express — высокоскоростной шины обмена данными между процессором и видеокартой).

В Zenith мы внедрили класс `MeshPool` — **Глобальный пул мешей**.

```
  ==========================================================
  |                 MeshPool VRAM (1.5 GB)                 |
  +----------------------------------------------------------+
  |  +----------+  +----------+  +----------+              |
  |  | Chunk 1  |  | Chunk 2  |  | Chunk 3  |   ... Свобод |
  |  | (Alloc)  |  | (Alloc)  |  | (Alloc)  |   ... Память |
  |  +----------+  +----------+  +----------+              |
  ==========================================================
   Один гигантский VBO/EBO на всю игру. 0 переключений буферов.
```

### Как это устроено?
При старте игры движок Zenith резервирует в видеопамяти один гигантский кусок размером **1.5 ГБ**.
Когда чанк сгенерировал свои блоки:
1. Вместо создания нового буфера `new VBO()` он просит свободный кусочек у нашего менеджера `MeshPool`.
2. Менеджер находит свободную ячейку нужного размера (как диспетчер на огромном складе).
3. Копирует вершины чанка прямо в этот глобальный буфер по определенному смещению.
4. Возвращает чанку простую структуру `Mesh`, где записано: *«Твои вершины начинаются с индекса №10405, а треугольники — с индекса №5023»*.

Во время рендеринга мы привязываем глобальный VAO (Vertex Array Object — объект массива вершин, хранящий настройки атрибутов) нашего пула **всего один раз за кадр**. Больше никаких переключений буферов.

## 2.5 Global Data UBO (std140): Общая доска объявлений на GPU

### В чём проблема?
Чтобы нарисовать любой объект, видеокарте нужно знать кучу общих вещей: «Где камера?», «Где солнце?», «Какое сейчас время?». Раньше процессор перед отрисовкой каждого блока вызывал тяжёлые функции `glUniformMatrix4` и `glUniform3f`. В кадре тысячи объектов. В итоге процессор захлёбывался в бесконечных звонках видеокарте по пустякам.

### Метафора из жизни: Доска объявлений в офисе
Вместо того чтобы директор компании (`CPU`) подбегал лично к каждому сотруднику (`шейдеру`) и шептал на ухо: *"Слушай, время 12:00, курс доллара 90, обед в час"*, он просто вывешает красивый печатный плакат на **общую доску объявлений** (`Uniform Buffer Object` или UBO — буфер для глобальной передачи параметров в видеокарту). 
Все шейдеры молча подходят к этой доске и берут нужные данные. Процессор пишет туда данные **ровно один раз за кадр**.

### GLSL (OpenGL Shading Language — язык программирования шейдеров) Структура UBO «на пальцах»

```glsl
// layout std140 — это строжайший контракт! Он гарантирует, что байты в памяти 
// видеокарты будут лежать в том же порядке, в каком Java записала их в буфер.
layout (std140, binding = 0) uniform GlobalData {
    mat4 uProjection;   // 64 байта: Матрица перспективы (превращает 3D в 2D)
    mat4 uView;         // 64 байта: Матрица камеры (поворот и позиция зрителя)
    vec3 uCameraPos;    // 12 байт + 4 байта PADDING! Внимание, тут ловушка!
    vec3 uSunDirection; // 12 байт + 4 байта PADDING! Опять выравнивание!
    float uTime;        // 4 байта: Текущее время игры (для анимаций)
};
```

> [!WARNING]
> **Ловушка выравнивания std140!**
> Стандарт `std140` выравнивает базовые типы по правилу кратности 16 байтам (размер `vec4`). Это значит, что трехкомпонентный `vec3 uCameraPos` в памяти GPU займёт не 12 байт, а **16 байт** (12 байт данных + 4 байта «мусорного» отступа-padding). Если в Java-коде при заполнении `ByteBuffer` вы запишете `uSunDirection` сразу после 12 байт `uCameraPos` без учёта этого отступа, все координаты «поплывут», и солнце улетит в другое измерение.

## 3. MultiDraw Indirect (MDI — механизм массовой отрисовки множества объектов одной командой): Как нарисовать мир за один вызов

### В чём проблема?
Классический рендеринг чанков выглядит так:
* CPU: *"Эй, видеокарта, нарисуй чанк №1!"* (Вызов Draw Call)
* GPU: *"Рисую..."*
* CPU: *"А теперь чанк №2!"* (Draw Call)
* GPU: *"Рисую..."*
Если дальность прорисовки большая, процессор отправляет тысячи таких команд за кадр. Шина передачи данных между CPU и GPU перегружается, FPS проседает, хотя видеокарта загружена всего на 10%.

### Метафора из жизни: Огромный грузовик вместо курьеров на самокатах
В **Zenith** мы перешли на GPU-Driven подход. 
Весь наш мир хранится в одном гигантском массиве видеопамяти — `MeshPool` (объемом до 1.5 ГБ VRAM).
Вместо тысяч мелких посылок мы загружаем один огромный грузовик. А процессору остается лишь составить **"путевой лист"** (Indirect Buffer) — массив простых структур команд, где написано: *"Начать рисовать с вершины №500, взять 300 индексов, это чанк А. Затем с вершины №800 взять 400 индексов, это чанк Б"*.

После чего CPU делает всего **один-два системных вызова** `glMultiDrawElementsIndirect`. Видеокарта забирает путевой лист и сама параллельно отрисовывает весь мир за микросекунды.

Каждая строчка в нашей книге-плане (**DrawElementsIndirectCommand**) весит ровно 20 байт и состоит из 5 целых чисел:

```glsl
struct DrawElementsIndirectCommand {
    uint count;         // Сколько индексов треугольников нарисовать (например, 36)
    uint instanceCount; // Сколько копий нарисовать (для чанков обычно 1)
    uint firstIndex;    // С какого индекса в глобальном EBO начать чтение
    uint baseVertex;    // С какого индекса в глобальном VBO начать чтение вершин
    uint baseInstance;  // Смещение в буфере данных инстансов
};
```

### Как работает оркестратор `MultiDrawBatch`?
1. В каждом кадре система `ChunkRenderSystem` определяет, какие чанки видны игроку (отсекая те, что находятся в темных пещерах за стенами).
2. Для каждого видимого чанка мы вызываем `batch.put(mesh)`.
3. Наш оркестратор `MultiDrawBatch` записывает 20-байтную команду `DrawElementsIndirectCommand` во временный буфер в оперативной памяти Java.
4. В конце кадра `MultiDrawBatch` одним махом копирует весь массив команд в VRAM видеокарты в специальный буфер команд **`GL_DRAW_INDIRECT_BUFFER`**.
5. Процессор делает один-единственный вызов:

```java
public void drawBatchedChunks(int chunkCount) {
    // 1. Привязываем VAO нашего гигантского MeshPool
    // Больше никаких переключений буферов в цикле!
    glBindVertexArray(meshPool.getVaoId());
    
    // 2. Привязываем наш "путевой лист" — буфер команд отрисовки
    glBindBuffer(GL_DRAW_INDIRECT_BUFFER, indirectCommandBufferId);
    
    // 3. Отдаем ОДНУ команду графическому чипу!
    glMultiDrawElementsIndirect(
        GL_TRIANGLES,           // Рисуем треугольниками
        GL_UNSIGNED_INT,        // Индексы упакованы в 32-битные целые числа
        0,                      // Начинаем читать команды с начала буфера
        chunkCount,             // Сколько чанков нарисовать за этот присест
        0                       // Шаг между командами в байтах (0 = лежат вплотную)
    );
    
    // Готово. Процессор свободен и может заниматься физикой сущностей.
}
```

## 4. Битовая упаковка вершин Standard Mesh: сжимаем геометрию в 2 раза

Чтобы видеопамять не забивалась терабайтами геометрии, вершина в Zenith сжата до **28 байт** (вместо стандартных 48-64 байт у большинства движков).

Один стандартный тип `float` в Java занимает 4 байта (32 бита). Но для многих параметров блока (например, нормали, которая принимает всего 6 значений от 0 до 5) выделять 32 бита — чистое расточительство.

Мы упаковываем несколько значений в биты одного-единственного целого числа `uint` (32 бита) с помощью битовых сдвигов.

```
        ==========================================================
        |                 Число PackedData0 (32 бита)            |
        +-----------------------+----------------+---------------+
        |    Координата X       |  Координата Y  | Координата Z  |
        |      (10 бит)         |    (10 бит)    |   (10 бит)    |
        ==========================================================
```

### Распаковка упакованных вершин в GLSL-шейдере (`vertex.glsl`):
```glsl
#version 330 core

// Принимаем всего два 32-битных числа вместо кучи float-переменных!
layout(location = 0) in uint aPackedData0;
layout(location = 1) in uint aPackedData1;

out vec3 vNormal;
out vec3 vUVW;
out float vWindWeight;

// Заранее заготовили массив из 6 возможных нормалей воксельного куба
const vec3 NORMALS[6] = vec3[](
    vec3( 0,  1,  0), // 0: Вверх
    vec3( 0, -1,  0), // 1: Вниз
    vec3( 1,  0,  0), // 2: Вправо
    vec3(-1,  0,  0), // 3: Влево
    vec3( 0,  0,  1), // 4: Вперед
    vec3( 0,  0, -1)  // 5: Назад
);

void main() {
    // 1. Распаковываем координаты X, Y, Z (каждая занимает по 10 бит)
    float x = float(aPackedData0 & 0x3FFu) / 16.0;         // Бит 0-9
    float y = float((aPackedData0 >> 10) & 0x3FFu) / 16.0; // Бит 10-19
    float z = float((aPackedData0 >> 20) & 0x3FFu) / 16.0; // Бит 20-29
    vec3 localPos = vec3(x, y, z);
    
    // 2. Распаковываем текстурные координаты (u, v) и индекс слоя массива текстур (w)
    float u = float(aPackedData1 & 0x1Fu) / 16.0;         // Бит 0-4
    float v = float((aPackedData1 >> 5) & 0x1Fu) / 16.0;  // Бит 5-9
    float w = float((aPackedData1 >> 10) & 0x3Fu);        // Бит 10-15 (Слой текстурного атласа)
    vUVW = vec3(u, v, w);
    
    // 3. Распаковываем нормаль куба
    uint normalIdx = (aPackedData1 >> 16) & 0xFFu;        // Бит 16-23
    vNormal = NORMALS[normalIdx];
    
    // 4. Распаковываем вес колыхания ветра (для травы и листвы)
    vWindWeight = float((aPackedData1 >> 24) & 0xFFu) / 100.0; // Бит 24-31
    
    gl_Position = uProjectionView * vec4(localPos, 1.0);
}
```
Итого: мы передаём всего два числа `uint`, а видеокарта на лету за доли такта распаковывает из них полные 3D-координаты, текстуры, нормали и веса ветра. Расход видеопамяти сократился вдвое.

> [!NOTE]
> **Магия битовых масок для скрипт-разработчиков**
> Представьте, что 32-битное число — это гирлянда из 32 лампочек. Нам нужно прочитать только гирлянду с 10-й по 19-ю лампочку (это наша координата Y). 
> 1. Сдвиг `>> 10` перетаскивает эти нужные лампочки в самое начало гирлянды.
> 2. Маска `& 0x3FFu` (где `0x3FF` — это 10 горящих лампочек подряд, т.е. число 1023) работает как картонный трафарет: она накладывается поверх гирлянды и пропускает свет только от первых 10 лампочек, отсекая весь остальной мусор. Так мы получаем чистое значение Y!

## 5. Connected Textures, Smooth Lighting & Shader State Guards (Графический AAA-шик)

Воксельные кубики без сглаживания выглядят плоско и дёшево. Чтобы сделать мир Zenith органичным, бесшовным и приятным для глаз, мы реализовали три мощных графических алгоритма.

### 5.1 Connected Textures / CTM (Соединение текстур)
Когда рядом стоят одинаковые блоки (например, стеклянные окна или каменная кладка), мы не хотим видеть между ними глупые внутренние швы рамы или стыки текстур. Они должны сливаться в единую монолитную конструкцию.

*   **Как это работает в Zenith:**
    1.  **Опрос соседей:** При генерации меша чанка движок для каждого блока опрашивает 8 соседних блоков, лежащих на плоскости его грани.
    2.  **Битовая маска:** Формируется **8-битная маска смежности (0-255)**. Каждый бит равен `1`, если соседний блок — такой же (например, тоже стекло), и `0`, если другой.
    3.  **Индекс текстуры:** По этой маске в специальной таблице LUT (Look-Up Table) находится индекс нужного спрайта. Всего для идеального бесшовного соединения углов и бортиков нужно 47 уникальных текстурных сегментов!
    4.  **Shader-side UV Offset:** В шейдере мы просто передаём индекс сегмента, и фрагментный шейдер автоматически смещает текстурные координаты, подставляя нужный уголок, рамку или чистое стекло. Соединения получаются идеально мягкими.

### 5.2 Dynamic Smooth Lighting & Ambient Occlusion (Мягкие тени углов)
В реальной жизни углы комнат или стыки стен всегда темнее, потому что туда долетает меньше рассеянного света. Это называется **Ambient Occlusion (AO)**.

Если освещать весь воксельный куб целиком одним числом, игра будет выглядеть плоской. 
В Zenith мы рассчитываем освещенность индивидуально для **каждого из 4 углов каждой грани**:

```
        +---------------+---------------+
        |  Соседний B1  |  Соседний B2  |
        |  (Свет = 12)  |  (Свет = 8)   |
        +---------------+---------------+
        |               | Вершина V (AO)| <-- Считаем свет в этом углу!
        |               | (Тут стык)    |
        +---------------+---------------+
         Итоговый свет в углу V = (Свет_Блока + B1 + B2 + Угловой_Блок) / 4
```

Мы берём среднее арифметическое освещённости трёх соседних блоков, образующих этот угол, и записываем его в вершины. Видеокарта автоматически интерполирует (плавно сглаживает) освещённость по всей грани блока. На экране возникают реалистичные мягкие тени в углах.

#### Защита от вспышек света (AO Breaking Guards)
Здесь нас подстерегал коварный баг. Когда игрок бьёт кирочкой по блоку камня, блок начинает разрушаться, но физически он всё ещё стоит в мире. При этом система генерации меша чанка постоянно перестраивает геометрию ломаемого блока, чтобы наложить текстуру трещин.

Если соседний чанк при генерации AO решит, что ломаемый блок уже исчез (ведь он в процессе разрушения), он зальёт углы стен ярким солнечным светом. Результат: при каждом ударе по блоку соседние стены начинают мерцать белыми вспышками.

**Решение в Zenith:**
Мы внедрили механизм **AO Breaking Guards**. В генераторе меша `ChunkNeighborhood.java` при сглаживании света мы проверяем позицию ломаемого блока через `world.getBreakingPos()`. 
Если координата соседа совпадает с ломаемым блоком, алгоритм принудительно подменяет его тип:
```java
// Внутри ChunkNeighborhood.java при расчете AO и Smooth Light
if (neighborPos.equals(world.getBreakingPos())) {
    // Принудительно считаем ломаемый блок ПЛОТНЫМ и НЕПРОЗРАЧНЫМ со стороны соседа,
    // блокируя солнечный свет и намертво устраняя вспышки и мерцание стен при добыче!
}
```
Это дало идеально стабильную картинку при майнинге.

### 5.3 OpenGL State Leak Guards (Менеджер состояний шейдеров)
Когда мы рендерим сцену, мы запускаем разные независимые проходы рендеринга (`RenderPass`): сначала рисуем чанки, потом оверлей ломания блоков с анимацией тряски (wobble), потом 3D-обводку выбранного блока.

В OpenGL все настройки шейдеров глобальны. И если в проходе обводки или ломания блока мы установили в шейдер флаг тряски `uIsProxy = true` или амплитуду качания листвы `uSwayOverride = 1.2f`, то после окончания прохода эти значения **останутся в видеокарте**!
В следующем кадре при рисовании обычного мира эти «утекшие» настройки применятся к обычным блокам. Листва начнёт хаотично замирать, а стены чанков — ложно деформироваться и трястись от ветра вместе с травой. Это называется **State Leak (утечка состояния)**.

**Решение в Zenith:**
Мы написали централизованный менеджер **`ShaderStateManager`**.
Перед стартом каждого прохода рендеринга и после его завершения мы вызываем метод `resetToBaseline()`. Он принудительно сбрасывает все опасные юниформы видеокарты в исходные безопасные значения:
```java
// com.za.zenith.engine.graphics.ShaderStateManager.java
public static void resetToBaseline() {
    activeShader.setVector4f("uBreakingPos", ZERO_VECTOR);
    activeShader.setFloat("uSwayOverride", -1.0f); // Выключить переопределение ветра
    activeShader.setBoolean("uIsProxy", false);    // Выключить wobble-тряску
    // Сброс текстурных слоев и масок...
}
```
Благодаря этому менеджеру графические системы Zenith работают в полной изоляции, исключив любые визуальные глитчи и мерцания.

### 5.4 Texture Arrays: Избавляемся от швов коммунальных атласов

#### Метафора из жизни: ЖК "Массивы текстур" вместо коммунальной квартиры
Представьте огромную коммунальную квартиру, где все жильцы делят одну гигантскую простыню. Это классический 2D-атлас текстур, где все блоки игры нарисованы на одном большом холсте.

В чём проблема такого общежития? Когда камера отдаляется от блока, видеокарта пытается сгладить его картинку (строит мип-мапы) и начинает невольно захватывать пиксели соседних текстур на атласе. В итоге на стыках блоков в мире появляются цветные полосы — «швы» (edge bleeding), а на расстоянии всё начинает мерцать и рябить.

Zenith использует **Texture Arrays (GL_TEXTURE_2D_ARRAY)** — современный аналог этого «общежития». Теперь у каждой текстуры блока есть свой собственный изолированный «этаж» (слой массива). Видеокарта строит мип-мапы для каждого слоя строго отдельно. Границы цветов защищены физически — им просто не с кем смешиваться. Чтобы указать видеокарте на нужный пиксель, мы используем трёхмерные текстурные координаты (U, V, W), где компонента W — это номер нужного этажа-слоя в массиве.

#### Код «на пальцах»
Посмотрим, насколько элегантным становится фрагментный шейдер блоков при использовании массивов текстур:

```glsl
// fragment.glsl
#version 330 core

in vec3 v_TexCoordUVW;     // 3D-координаты: U, V и W (индекс слоя)
in float v_LightIntensity; // Интенсивность освещения блока

out vec4 f_Color;          // Итоговый цвет пикселя на экране

// Вместо обычного sampler2D мы используем массив 2D текстур!
uniform sampler2DArray u_TextureArray;

void main() {
    // Вытаскиваем цвет пикселя из массива текстур. 
    // Графический чип сам знает, как отсэмплировать нужный слой v_TexCoordUVW.z
    // Без швов, без размытия соседних текстур, идеально четко!
    vec4 texColor = texture(u_TextureArray, v_TexCoordUVW);
    
    // Если пиксель прозрачный (например, дырка в листве), выбрасываем его из рендеринга
    if (texColor.a < 0.1) {
        discard;
    }
    
    // Применяем освещение к цвету текстуры
    f_Color = vec4(texColor.rgb * v_LightIntensity, texColor.a);
}
```

### 5.5 Smart Face Culling: Экономим ресурсы с умом

#### Метафора из жизни: Не крась скрытые кирпичи
Представьте, что вы строите кирпичную стену дома и хотите сэкономить на дорогой краске. Красить внутренние стороны кирпичей, которые плотно прижаты друг к другу, никто не станет — их никогда не увидят. Видеокарта в играх рассуждает так же: она отбрасывает (cull-ит) невидимые грани блоков, экономя значительное количество видеопамяти.

Но что произойдёт, если блок не является идеальным цельным кубом 1x1x1? Например, это глиняный кувшин или тонкая деревянная плита. Если мы применим стандартное слепое отбрасывание, то при размещении кувшина вплотную к стене его задняя стенка просто исчезнет, открыв портал в пустоту.

В **Zenith** мы внедрили **Smart Face Culling**. Наша система скрывает грань блока только в одном случае: если блоки соприкасаются по *абсолютно полной* плоскости 1x1x1 И соседний блок является стопроцентно непрозрачным. Во всех остальных случаях (щели, полублоки, декорации) грани отрисовываются честно, сохраняя целостность и красоту мира.

#### Код «на пальцах»
Посмотрим на логику принятия решений в коде сборщика мешей чанков:

```java
public class BlockCuller {
    /**
     * Определяет, нужно ли рендерить конкретную грань блока на границе с соседом
     */
    public static boolean shouldRenderFace(Block block, Block neighbor, Direction direction) {
        // 1. Если соседа нет (там воздух) — грань однозначно нужно рисовать
        if (neighbor == null || neighbor.isAir()) {
            return true;
        }
        
        // 2. Если сосед пропускает свет (стекло, листва) — грань обязана быть видна сквозь него
        if (neighbor.isTransparent()) {
            return true;
        }
        
        // 3. Smart Face Culling в действии!
        // Извлекаем сложные физические формы (набор ограничивающих боксов AABB — Axis-Aligned Bounding Box, выровненных по осям координат) обоих блоков
        VoxelShape myShape = block.getShape();
        VoxelShape neighborShape = neighbor.getShape();
        
        // Проверяем: закрывает ли сосед плоскость соприкосновения полностью?
        // Метод isFaceFull проверяет, занимает ли форма блока всю площадь грани (1x1)
        boolean isFaceFullyCovered = neighborShape.isFaceFull(direction.getOpposite()) 
                                     && myShape.isFaceFull(direction);
        
        // Если оба блока полностью соприкасаются этой плоскостью — прячем грань (false).
        // Если есть хоть малейший зазор (плита, ступенька, кувшин) — рисуем её (true)
        return !isFaceFullyCovered;
    }
}
```

## 5.6 Classic Voxel Particle System: Билборды на GPU и Умные Крошки (UV Snippets)

### Метафора из жизни: Колыхание щепок в воздухе
Если бы компьютер честно симулировал каждую микро-крошку как полноценный 3D-объект с тысячей полигонов и коллизиями, видеокарта расплавилась бы моментально.

Решение — **GPU Billboarding**. Мы рисуем абсолютно плоские 2D-картинки крошек, но заставляем вершинный шейдер мгновенно разворачивать эти картинки лицом к камере, с какого бы ракурса на них ни смотрели. Для игрока они всегда кажутся объёмными. Чтобы они не застревали в стенах и не вызывали баги с мерцанием, физические коллизии у них полностью убраны — они летят по параболе гравитации и плавно растворяются в воздухе.

Но как сделать так, чтобы отколовшаяся крошка от камня выглядела как камень, а от дерева — как дерево? Если просто сжать текстуру целого блока и натянуть её на песчинку, получится нелепое мыло. Здесь работает **Snippet UV Mapping**. Рендерер вырезает случайный крошечный кусочек (размером 4×4 пикселя) прямо из оригинальной текстуры блока (16×16 пикселей). В итоге плотность пикселей сохраняется идеальной (1:1), а крошки выглядят как настоящая текстурная пыль.

### Код «на пальцах»
Заглянем внутрь вершинного шейдера частиц и посмотрим, как это происходит на GPU:

```glsl
// Вершинный шейдер частиц (particle_vertex.glsl)
#version 330 core
layout (location = 0) in vec3 a_Position; // Вершина плоского квада (от -0.5 до 0.5)
layout (location = 1) in vec2 a_TexCoord; // Базовые UV-координаты квада (0..1)

// Инстансинг: эти данные уникальны для каждой отдельной частицы
layout (location = 2) in vec3 i_Position;   // Где в мире находится центр частицы
layout (location = 3) in float i_Roll;      // Угол вращения частицы вокруг своей оси
layout (location = 4) in float i_Scale;     // Насколько частица большая
layout (location = 5) in float i_Alpha;     // Коэффициент прозрачности
layout (location = 6) in float i_TexLayer;  // На каком слое массива текстур лежит блок
layout (location = 7) in vec2 i_SnippetOffset; // Смещение нашей 4x4 крошки в текстуре

uniform mat4 u_View;       // Матрица камеры (куда мы смотрим)
uniform mat4 u_Projection; // Матрица проекции (как мы видим перспективу)

out vec3 v_TexCoordUVW;    // Отправляем во фрагментный шейдер 3D-координаты
out float v_Alpha;

void main() {
    // 1. Извлекаем направление "право" и "верх" камеры прямо из матрицы вида!
    // Первые три элемента первой строки — вектор "вправо", второй строки — "вверх".
    // Это заставляет плоский квадрат всегда стоять лицом к объективу!
    vec3 cameraRight = vec3(u_View[0][0], u_View[1][0], u_View[2][0]);
    vec3 cameraUp    = vec3(u_View[0][1], u_View[1][1], u_View[2][1]);

    // 2. Крутим нашу частицу! Вращаем 2D-координаты вершины с помощью школьной тригонометрии
    float cosR = cos(i_Roll);
    float sinR = sin(i_Roll);
    vec2 rotatedPos = vec2(
        a_Position.x * cosR - a_Position.y * sinR,
        a_Position.x * sinR + a_Position.y * cosR
    );

    // 3. Собираем финальную мировую позицию частицы: 
    // Берём её центр в пространстве и смещаем влево/вправо и вверх/вниз относительно взгляда камеры!
    vec3 worldPos = i_Position + 
                    (cameraRight * rotatedPos.x * i_Scale) + 
                    (cameraUp * rotatedPos.y * i_Scale);

    // Переводим всё это дело в экранные координаты
    gl_Position = u_Projection * u_View * vec4(worldPos, 1.0);

    // 4. Отрезаем кусочек текстуры!
    // Поскольку размер крошки 4x4 на текстуре 16x16 — это ровно 1/4 (или 0.25) её размера,
    // мы сжимаем стандартные координаты (0..1) в четыре раза и сдвигаем их на наш случайный оффсет.
    vec2 snippetUV = i_SnippetOffset + (a_TexCoord * 0.25);

    // Передаем 3D-вектор: U, V и индекс текстурного слоя W
    v_TexCoordUVW = vec3(snippetUV, i_TexLayer);
    v_Alpha = i_Alpha;
}
```

## 6. Готовимся к IT-собеседованию

### Вопрос 1: В чем разница между CPU-Driven и GPU-Driven архитектурой рендеринга?
* **Ответ:** В CPU-Driven директор (процессор) лично командует отрисовкой каждого объекта, перегружая PCIe шину и драйвер тысячами звонков `glDrawElements` (Draw Call Bottleneck). В GPU-Driven процессор лишь один раз сгружает все данные в глобальный буфер и пишет «книгу-план» команд. Видеокарта считывает этот план из своей памяти и отрисовывает всю сцену за один вызов (`glMultiDrawElementsIndirect`), полностью разгружая процессор.

### Вопрос 2: Зачем нужна система версионирования в Mesh Pool (`poolVersion`)? Что будет, если её отключить?
* **Ответ:** Наш Mesh Pool работает по кругу (Wrap-around): когда свободное место заканчивается, он начинает перезаписывать новые чанки в самое начало, затирая старые. Система версионирования отслеживает поколения буфера.
* **Что сломается:** Если мы её отключим, может случиться так, что старый чанк все еще стоит в очереди на рендеринг, а его вершины в видеопамяти уже перезаписаны новым сгенерированным чанком. Видеокарта считает некорректные координаты, и на экране возникнет жуткий визуальный баг — растянутые треугольники на пол-экрана («визуальная каша»).

### Вопрос 3: Зачем нужна битовая упаковка вершин вокселей и как она распаковывается?
* **Ответ:** Чтобы экономить драгоценную видеопамять (VRAM) и пропускную способность PCIe. Вместо отправки кучи дробных чисел `float` (каждое по 32 бита), мы упаковываем несколько маленьких параметров (координаты блока 0-16, нормали 0-5, текстуры) в биты всего двух целых чисел `uint` с помощью битовых сдвигов `<<` в Java. В GLSL-шейдере видеокарта на лету за доли такта распаковывает их обратно сдвигами `>>` и масками `&`.
