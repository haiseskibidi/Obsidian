## 1. Skeletal Viewmodel (модель от первого лица): Как оживить руки без шейдерного насилия

### Суть проблемы на пальцах
Вспомните деревянных кукол-марионеток. Если у куклы вместо суставов гвозди, её руки двигаются как палки шлагбаума. Но в реальности, когда вы тянетесь за чашкой кофе, плечо и локоть совершают микродвижения вслед за кистью.

В **Zenith** рендеринг рук от первого лица переведён на полноценную скелетную иерархию костей: `Shoulder` (плечо) $\to$ `Forearm` (предплечье) $\to$ `Hand` (кисть). Вместо того чтобы нагружать видеокарту тяжёлым шейдерным скиннингом (который высчитывает веса вершин для каждого вокселя), применён трюк с распределением углов. При резком повороте головы (камеры) кисть делает основной поворот (70% от угла), локоть слегка сгибается (20%), а плечо доворачивается на оставшиеся 10%. Рука выглядит «живой» и мягкой, а не приваренной к камере рельсой.

Чтобы при экстремальных изгибах суставы не распадались на висящие в воздухе куски, геометрия воксельных костей в JSON смоделирована с небольшим нахлёстом. Они заходят друг в друга, как колена выдвижной подзорной трубы или элементы рыцарского доспеха. Никаких дыр в локтях.

### Разбор кода «на пальцах»
Посмотрим на Java-метод, который строит локальные матрицы трансформации костей скелета:

```java
/**
 * Вычисляет мировую матрицу трансформации для конкретной кости скелета
 * с учетом иерархии и распределения углов поворота камеры.
 */
public void updateBoneTransform(Matrix4f parentTransform, float camYaw, float camPitch) {
    // 1. Получаем индивидуальную долю влияния камеры на сустав.
    // Для плеча (Shoulder) это 0.10f, для локтя (Forearm) - 0.20f, для кисти (Hand) - 0.70f.
    float influence = this.getRotationInfluence();

    // 2. Создаем локальную матрицу трансформации кости.
    Matrix4f localMatrix = new Matrix4f()
        // Смещаем начало координат в точку сустава (точку вращения кости)
        .translation(this.localOffset)
        // Применяем базовую анимацию покачивания из JSON-трека
        .rotate(this.localAnimationRotation)
        // Поворачиваем сустав вслед за взглядом игрока по горизонтали (Y) и вертикали (X)
        .rotateY(camYaw * influence)
        .rotateX(camPitch * influence);

    // 3. Умножаем локальную матрицу на матрицу родительской кости.
    // Матрица предплечья помнит, где находится плечо, а кисть "наследует" положение предплечья!
    this.worldMatrix.set(parentTransform).mul(localMatrix);

    // 4. Рекурсивно передаем эстафету "дочерним" суставам
    for (Bone child : this.children) {
        child.updateBoneTransform(this.worldMatrix, camYaw, camPitch);
    }
}
```

## 2. 2nd-Order Spring Physics: Физика пружин второго порядка для инерции рук

### Суть проблемы на пальцах
В старых шутерах оружие в руках персонажа казалось приклеенным к экрану. При резком повороте головы оно двигалось абсолютно синхронно с камерой — выглядит фальшиво.

Чтобы руки в **Zenith** ощущались живыми и весомыми, они привязаны к камере не жестко, а через виртуальную **физическую пружину второго порядка** (пружина-масса-демпфер). 
- Если в руках массивный каменный молот (`weight = 5.0` в JSON), пружина становится «мягкой» и податливой: молот плавно отстает при взмахе камеры и тяжело покачивается при остановке.
- Если в руках легкий нож или травинка (`weight = 0.2`), пружина натягивается до предела, мгновенно передавая движение взгляда на экран.
А чтобы при быстрой смене предметов в инвентаре оружие не улетало в космос от накопленной скорости, метод `reset()` мгновенно обнуляет скорость физической пружины и ставит её в целевую позицию.

### Метафора из жизни
Представьте, что вы держите в руке тяжёлую чугунную сковородку. При резком повороте на 90 градусов сковорода повернётся не сразу — отстанет от движения. А когда вы остановитесь, она по инерции пролетит чуть дальше, качнётся пару раз (перелёт/overshoot) и только потом замрёт.

### Разбор кода «на пальцах»
Численное интегрирование пружины второго порядка методом полунеявного Эйлера:

```java
public class InertiaSpring {
    private float position = 0.0f; // Текущая позиция (сдвиг) руки на экране
    private float velocity = 0.0f; // Текущая скорость смещения руки

    // Коэффициенты системы уравнений
    private float f; // Жесткость пружины (естественная частота колебаний)
    private float z; // Коэффициент амортизации (демпфер, гасящий колебания)
    private float r; // Инерционный толчок (коэффициент отставания)

    /**
     * Пересчитывает физические характеристики пружины на основе веса предмета
     */
    public void configureForItem(float weight) {
        // Чем тяжелее предмет, тем меньше частота колебаний 'f' (пружина становится мягче!)
        this.f = 6.0f / (1.0f + weight); 
        this.z = 0.75f; // Оптимальный демпфер: рука почти не совершает паразитных колебаний
        this.r = -0.6f; // Коэффициент запаздывания при резком старте
    }

    /**
     * Обновляет состояние пружины за шаг времени dt.
     * target - целевая координата, куда стремится рука (например, центр экрана)
     */
    public float update(float target, float dt) {
        // Рассчитываем константы для дифференциального уравнения
        float k1 = z / ((float)Math.PI * f);
        float k2 = 1.0f / ((2.0f * (float)Math.PI * f) * (2.0f * (float)Math.PI * f));
        float k3 = r * z / (2.0f * (float)Math.PI * f);

        // Сила упругости пружины, пропорциональная расстоянию до цели
        float force = target - position;

        // Вычисляем ускорение согласно дифференциальному уравнению 2-го порядка
        // a = (force - k1 * v) / k2
        float acceleration = (force - k1 * velocity) / k2;

        // Интегрируем скорость и позицию (схема Эйлера-Кромера)
        velocity += acceleration * dt;
        position += velocity * dt;

        return position;
    }

    /**
     * Мгновенно переносит руку в цель и гасит скорость (нужно при смене предметов)
     */
    public void reset(float target) {
        this.position = target;
        this.velocity = 0.0f;
    }
}
```

## 3. HeldItemRenderer: Умное позиционирование блоков в руке

### Суть проблемы на пальцах
Если бы рендерер привязывал все блоки за их геометрический центр, то крупный промышленный чан пролезал бы сквозь предплечье до локтя, а тонкая декоративная пластина парила бы над ладонью.

`HeldItemRenderer` решает эту проблему элегантным геометрическим сдвигом. Он запрашивает размеры коллизии (Bounding Box / AABB — выровненного по осям параллелепипеда) блока, находит нижнюю и заднюю плоскости и сдвигает матрицу рендеринга так, чтобы блок буквально «стоял» на ладони своей нижней гранью и «прижимался» задней гранью к воображаемой перчатке игрока.

### Метафора из жизни
Возьмите в руку спичечный коробок — пальцы держат его с боков, задняя грань ложится в ладонь. Теперь возьмите коробку от обуви. Кисть осталась на том же месте, но сама коробка сместилась вперёд и вверх, чтобы не провалиться сквозь запястье.

### Разбор кода «на пальцах»
Смещение геометрии блока перед отрисовкой в руке:

```java
/**
 * Сдвигает матрицу трансформации блока так, чтобы он лежал точно в ладони.
 */
public void alignBlockInHand(AABB bounds, Matrix4f modelViewMatrix) {
    // 1. Центрируем блок по оси X (чтобы он не съезжал влево или вправо от центра кисти)
    float xOffset = -(bounds.minX + bounds.maxX) / 2.0f;

    // 2. Ставим блок основанием на ладонь (ось Y)
    float yOffset = -bounds.minY;

    // 3. Сдвигаем по оси Z так, чтобы задняя грань блока (-Z) касалась поверхности ладони.
    // minZ - это координата задней стенки блока. Смещая модель на -minZ,
    // мы совмещаем заднюю грань с локальным нулем руки!
    float zOffset = -bounds.minZ;

    // Применяем рассчитанный вектор смещения к матрице вида
    modelViewMatrix.translate(xOffset, yOffset, zOffset);
}
```

## 4. PCA Engine Auto-Orientation: Умные глаза графического движка

### Суть проблемы на пальцах
Художники часто рисуют кирки или мечи по диагонали (чтобы они красиво смотрелись на иконке в инвентаре), а мешочки или зелья — строго вертикально. Чтобы не прописывать вручную углы поворота и смещения для каждого из сотен предметов, используется алгоритм PCA (Principal Component Analysis — метод главных компонент).

Он сканирует все непрозрачные пиксели на текстуре, считает их «центр масс» и определяет ковариационную матрицу — то есть понимает, вдоль какого направления пиксели сильнее всего «размазаны». На основе этого находится угол наклона главной оси. Затем движок автоматически поворачивает модель так, чтобы рукоятка кирки встала ровно, а точка хвата (Smart Grasping) привязывается к нижнему краю этой оси. Любой предмет ложится в руку игрока корректно — без единой строчки хардкода.

### Метафора из жизни
Представьте, что вам завязали глаза и дали в руки длинный предмет — веник или топор. Вы быстро проведёте руками по нему, определите, в какую сторону он вытянут (это его главная ось), нащупаете рукоятку на одном конце и возьмётесь за неё.

### Разбор кода «на пальцах»
Математика ковариации и поиска угла главной оси на Java:

```java
/**
 * Анализирует пиксели текстуры методом главных компонент (PCA)
 * и вычисляет угол наклона основной оси предмета.
 */
public float calculatePrincipalAxisAngle(ByteBuffer pixels, int width, int height) {
    float meanX = 0, meanY = 0;
    int opaqueCount = 0;

    // Шаг 1: Ищем геометрический центр масс (средние координаты силуэта)
    for (int y = 0; y < height; y++) {
        for (int x = 0; x < width; x++) {
            if (isOpaque(pixels, x, y, width)) {
                meanX += x;
                meanY += y;
                opaqueCount++;
            }
        }
    }
    if (opaqueCount == 0) return 0.0f; // Пустая текстура? Поворачивать нечего.
    
    meanX /= opaqueCount;
    meanY /= opaqueCount;

    // Шаг 2: Вычисляем ковариационную матрицу распределения пикселей
    float covXX = 0, covXY = 0, covYY = 0;
    for (int y = 0; y < height; y++) {
        for (int x = 0; x < width; x++) {
            if (isOpaque(pixels, x, y, width)) {
                // Вычисляем отклонение каждого пикселя от центра масс
                float dx = x - meanX;
                float dy = y - meanY;
                
                covXX += dx * dx;
                covXY += dx * dy;
                covYY += dy * dy;
            }
        }
    }

    // Шаг 3: Находим угол наклона с помощью обратного тангенса.
    // Формула из линейной алгебры: tan(2 * theta) = 2 * covXY / (covXX - covYY)
    float diff = covXX - covYY;
    if (Math.abs(diff) < 1e-5f) {
        // Если распределение симметрично по осям, возвращаем 45 градусов в зависимости от ковариации
        return covXY > 0 ? (float)Math.PI / 4.0f : -(float)Math.PI / 4.0f;
    }
    
    // atan2 возвращает точный угол наклона вектора максимальной дисперсии
    return 0.5f * (float)Math.atan2(2.0f * covXY, diff);
}
```

## 5. 3D-пикинг Oriented Bounding Box (Ray-OBB): Укрощаем Пространство

Во встроенном редакторе анимаций (**Animation Studio**, F8) можно кликать мышкой прямо по трёхмерным костям рук персонажа или деталям удерживаемого топора, чтобы выделять их и вращать.

Но как компьютер понимает, по какой именно кости кликнул игрок на плоском 2D-экране монитора? Для этого используется **Raycasting (Трассировка лучей)**. Мы выпускаем виртуальный луч из курсора мыши вглубь трехмерного мира.

### Ужас Oriented Bounding Box (OBB)
Если бы кости рук висели ровно по осям X, Y, Z, задача была бы тривиальной — пересечение луча с выровненной коробкой (AABB). Но руки согнуты, пальцы наклонены, топор повёрнут по кватерниону во все стороны.
Каждый куб модели становится **Oriented Bounding Box (Ориентированной коробкой, OBB)**.

Прямое пересечение луча с наклонной коробкой в 3D-пространстве — вычислительно дорогая задача. Решение — принцип относительности: **переносим сам луч в локальное пространство кости.**

Мы берем глобальную матрицу трансформации кости `p.globalMatrix`, которая содержит все её повороты, смещения и масштабы, и **инвертируем её**. Инвертированная матрица делает обратное действие: она возвращает любую точку пространства назад в начало координат $(0, 0, 0)$, выравнивая кость по осям!

Если умножить координаты луча на эту инвертированную матрицу, кость станет ровной (превратится в простой AABB), а луч развернётся в локальном пространстве под нужным углом. После этого применяется простой и быстрый алгоритм пересечения луча с осевыми плоскостями.

```java
// com.za.zenith.engine.graphics.ui.editor.animation.TransformController.java
public boolean pickPart3D(float mx, float my, int sw, int sh, AnimationEditorState state, List<ModelNode> allParts) {
    Vector3f ro = new Vector3f(), rd = new Vector3f();
    // 1. Генерируем 3D-луч из экранных координат мыши
    getRay(mx, my, sw, sh, state, ro, rd);
    
    ModelNode best = null; 
    float minD = Float.MAX_VALUE;

    for (ModelNode p : allParts) {
        if (p.def == null || p.def.cubes == null) continue;

        // 2. Инвертируем глобальную матрицу трансформации этой кости!
        Matrix4f invG = new Matrix4f(p.globalMatrix).invert();

        // 3. Умножаем луч на инвертированную матрицу! 
        // ro (Origin) умножается как Точка (Position), rd (Direction) - как Направление!
        Vector3f lro = ro.mulPosition(invG, new Vector3f());
        Vector3f lrd = rd.mulDirection(invG, new Vector3f());
        
        for (BoneDefinition.CubeDefinition c : p.def.cubes) {
            // Переводим размеры куба в метры (16 пикселей = 1 метр)
            float minX = c.x / 16.0f;
            float minY = c.y / 16.0f;
            float minZ = -(c.z / 16.0f + c.size[2] / 16.0f);
            
            Vector3f min = new Vector3f(minX, minY, minZ);
            Vector3f max = new Vector3f(min).add(new Vector3f(c.size[0]/16f, c.size[1]/16f, c.size[2]/16f));
            
            Vector2f res = new Vector2f();
            // 4. Пересекаем локальный луч с локальным AABB — без тяжёлой тригонометрии
            if (Intersectionf.intersectRayAab(lro, lrd, min, max, res)) {
                float t = res.x < 0 ? res.y : res.x;
                if (t > 0 && t < minD) { 
                    minD = t; 
                    best = p; 
                }
            }
        }
    }
    
    if (best != null) {
        state.selectedPart = best; // Кость выделена кликом мыши
        return true; 
    }
    return false;
}
```

## 6. Магнитное Притягивание предметов

Предметы в мире могут притягиваться к игроку при наличии компонента `MagneticComponent`. При расчете учитывается скорость игрока (Lead Pursuit — метод упреждения), что позволяет корректно подбирать предметы на ходу.

```java
if (isLockedOnPlayer) {

        player.getInventory().getActiveComponent(com.za.zenith.world.items.component.MagneticComponent.class);
    
    if (magnet != null) {
        float distance = (float)Math.sqrt(distSq);
        Vector3f dirToPlayer = new Vector3f(dx, dy, dz).normalize();
        
        // Квадратичное увеличение скорости сближения по мере приближения к игроку
        float approachSpeed = 12.0f + (1.0f - Math.min(1.0f, distance / 4.0f)) * (magnet.attractionForce * 0.2f);
        
        // Наследуем скорость игрока (Lead Pursuit) + притяжение
        velocity.set(player.getVelocity());
        velocity.add(dirToPlayer.mul(approachSpeed));
        onGround = false; // Отрываем предмет от земли
        
        // Визуальное вращение предмета в воздухе при полёте к игроку
        angularVelocity.y += deltaTime * 20.0f;
        angularVelocity.x += deltaTime * 5.0f;
    }
}
```

## 7. Mining Heat VFX (Visual Effects — визуальные эффекты): Раскаленные костяшки рук

При добыче ресурсов инструменты и руки игрока визуально раскаляются. Реализация `MiningVFXManager.java` предотвращает перенос эффекта нагрева между разными предметами.

```java
public class MiningVFXManager {
    private float handHeat = 0.0f;
    private float itemHeat = 0.0f;
    
    private int lastItemIdentity = -1;

    public void update(float deltaTime, Player player, MiningController mining) {
        ItemStack currentStack = player.getInventory().getSelectedItemStack();
        int currentIdentity = (currentStack != null) ? System.identityHashCode(currentStack) : 0;
        boolean hasItem = (currentStack != null && currentStack.getItem() != Items.HAND);

        // --- Сброс при смене предмета ---
        if (currentIdentity != lastItemIdentity) {
            itemHeat = 0.0f; // Новый взятый предмет всегда холодный
            if (mining.getBreakingBlockPos() != null) mining.reset();
        }
        lastItemIdentity = currentIdentity;

        float progress = (mining.getBreakingBlockPos() != null) ? mining.getBreakingProgress() : 0.0f;
        
        // Разделяем цели: нагреваем только то, чем сейчас бьем!
        float targetHand = (!hasItem) ? progress : 0.0f;
        float targetItem = (hasItem) ? progress : 0.0f;

        // Плавный разогрев (скорость 2.0) и медленное остывание (скорость 0.5)
        handHeat = updateChannel(handHeat, targetHand, deltaTime);
        itemHeat = updateChannel(itemHeat, targetItem, deltaTime);
    }

    private float updateChannel(float current, float target, float dt) {
        if (current < target) return Math.min(target, current + dt * 2.0f);
        return Math.max(target, current - dt * 0.5f);
    }
}
```

### Локализованное свечение во фрагментном шейдере
Вся модель не красится в красный цвет целиком. В шейдере рук используется маска весов костей: жар проявляется только на костяшках пальцев, где идёт физический удар. В шейдере предметов применяется вертикальная маска по Y — раскаляется только рабочая металлическая часть кирки/топора, а деревянная рукоятка остаётся холодной.
