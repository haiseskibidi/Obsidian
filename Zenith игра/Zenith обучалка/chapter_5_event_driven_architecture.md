# Глава 5. Реактивная шина событий & Архитектура ввода

В этой главе займёмся проектированием чистой архитектуры движка. Разберём, как избавиться от спагетти-кода и сделать системы ввода, звуков, частиц и логики независимыми друг от друга.

Рассмотрим, как устроена реактивная шина событий **EventBus** на примере городской доски объявлений, научимся отслеживать нажатия клавиш на лету без создания лишних объектов в памяти и разложим по полочкам алгоритм процедурных мишеней (**Weak Spots**) при добыче блоков.

---

## 1. Ужас сильной связанности (Tight Coupling)

Посмотрим на типичный код обработки нажатий клавиш в плохих игровых проектах:

```java
// Ужасный спагетти-код
public void handleKeySpace() {
    player.physics.jump();             // Надо прыгнуть!
    soundManager.playJumpSound();      // Надо издать звук!
    ui.staminaBar.reduce(10);          // Надо отнять стамину в UI (User Interface — пользовательском интерфейсе)!
    particleManager.spawnDust();       // Надо пустить пыль из-под ног!
}
```

Обработчик ввода знает абсолютно обо всём: о физике игрока, о звуковом менеджере, о вёрстке интерфейса стамины и о системе частиц пыли.

Если завтра художник захочет изменить цвет эффекта пыли или звукорежиссёр переименует файл прыжка, придётся лезть в класс ввода. Если нужно временно выключить UI стамины — этот код откажется компилироваться. Это **сильная связанность (Tight Coupling)**.

В Zenith мы разрубили этот гордиев узел с помощью двух паттернов:
1. **InputAction Layer** (Абстракция ввода).
2. **EventBus** (Реактивная шина событий).

## 2. Zero-Alloc InputAction: Битовые выключатели

Жёсткое правило: **игровая логика ничего не знает о физических клавишах клавиатуры.** Никакого прямого опроса GLFW-кодов (GLFW — Graphics Library Framework, библиотека для создания окон и обработки ввода) типа `GLFW_KEY_W` или `GLFW_KEY_SPACE`.

Вместо этого мы объявили перечисление логических действий `InputAction`:
`MOVE_FORWARD, JUMP, INTERACT, ATTACK, OPEN_JOURNAL`

Связь между физической клавишей и логическим экшеном задается в настройках (`options.json`) и может настраиваться игроком.

```
   Клавиатура (GLFW)         Битовый Массив (long)           Логический Экшен
  +-----------------+       +--------------------+         +-----------------+
  | Клавиша W зажата|------>| [0001000000000000] |-------->|  "move_forward" |
  | Клавиша Space   |       +--------------------+         |  "jump"         |
  +-----------------+        Битовые маски (long)          +-----------------+
```

### Как опрашивать кнопки без аллокаций?
Если на каждом кадре создавать объекты событий для нажатых клавиш, куча мгновенно забьётся мусором.

В `InputManager` мы применили битовую оптимизацию. Состояние всей клавиатуры и мыши хранится в трёх простых примитивных числах `long` (каждое по 64 бита):
* `pressedMask` — зажата ли клавиша прямо сейчас (кнопка удерживается).
* `justPressedMask` — была ли нажата строго в этом кадре (триггер клика).
* `justReleasedMask` — была ли отпущена строго в этом кадре.

Так как полезных кнопок в игре меньше 64 штук, **всё состояние клавиатуры и мыши идеально умещается в 3 числа.**

```java
// com.za.zenith.engine.input.InputManager.java
public class InputManager {
    private long pressedMask = 0L;
    private long justPressedMask = 0L;
    private long justReleasedMask = 0L;

    // Этот метод вызывается драйвером GLFW при любом клике по клавиатуре
    public void onKeyCallback(int keyId, int actionType) {
        // keyId - порядковый номер клавиши от 0 до 63
        long bit = 1L << keyId; // Сдвигаем единичку на нужный индекс бита
        
        if (actionType == GLFW.GLFW_PRESS) {
            pressedMask |= bit;        // Включаем тумблер (побитовое ИЛИ)
            justPressedMask |= bit;
        } else if (actionType == GLFW.GLFW_RELEASE) {
            pressedMask &= ~bit;       // Выключаем тумблер (побитовое И)
            justReleasedMask |= bit;
        }
    }

    // Мгновенный опрос состояния экшена в цикле
    public boolean isActionPressed(InputAction action) {
        int keyIndex = SettingsManager.getInstance().getBind(action);
        // Мгновенная битовая проверка, 0 аллокаций в куче
        return (pressedMask & (1L << keyIndex)) != 0L; 
    }
}
```
Итого: 0 объектов в памяти, 0 обращений к куче, а проверка нажатия выполняется процессором за микроскопическую долю наносекунды с помощью битовых масок.

> [!NOTE]
> **Побитовые операции «на пальцах»**
> Переменная `long` — это просто гирлянда ровно на 64 лампочки. 
> * Сдвиг `1L << keyId` берет одну включенную лампочку и сдвигает её на нужную позицию (например, 5-ю).
> * Операция побитового ИЛИ `|=` просто включает этот конкретный тумблер в ряду, не трогая остальные.
> * А чтобы проверить, зажата ли кнопка, мы используем побитовое И `&` (сравниваем нашу гирлянду с трафаретом). Если результат не ноль (`!= 0L`), значит нужная лампочка горит! Никаких массивов, списков и объектов — просто переключатели.

## 3. Реактивная шина событий: Городская доска объявлений

Чтобы подружить наши независимые системы (ввод, звуки, физику и частицы), мы используем класс `EventBus` (**Шина событий**), работающий по паттерну **Publish-Subscribe (Издатель-Подписчик)**.

Представьте городскую доску объявлений. 
Класс ввода при клике мышкой не бежит звонить физике и звукам лично. Он просто вешает на доску бумажку: *«Игрок кликнул ЛКМ (левой кнопкой мыши) по блоку»* (**BlockLeftClickEvent**).
На этом его работа полностью закончена.

```
                  ======================================
                  |         EventBus (Доска)           |
                  ======================================
                         │                       │
     Публикует записку   │                       │ Рассылает копии
     "Кликнули по блоку" │                       ▼ подписчикам
                  ========+=======      ========================
                  | InputHandler |      | - MiningController   |
                  ================      | - SoundManager       |
                                        | - ParticleManager    |
                                        ========================
```

Контроллеры-подписчики периодически подходят к доске, видят объявление и начинают делать свою работу:
1. **`MiningController`** видит записку, сверяет таймеры и наносит блоку урон.
2. **`ParticleManager`** видит ту же записку, берет тип блока и спавнит сочные щепки на экране.
3. **`SoundManager`** видит записку и проигрывает звук удара топора по дереву.

### Как это устроено в коде `EventBus.java`?
```java
public class EventBus {
    // Карта подписчиков, сгруппированная по классам событий
    private static final Map<Class<? extends Event>, List<Consumer<? extends Event>>> listeners = new ConcurrentHashMap<>();

    // Системы вызывают этот метод при запуске игры, чтобы подписаться на события
    public static <T extends Event> void subscribe(Class<T> eventClass, Consumer<T> listener) {
        listeners.computeIfAbsent(eventClass, k -> new CopyOnWriteArrayList<>()).add(listener);
    }

    // Вызывается для публикации события
    @SuppressWarnings("unchecked")
    public static <T extends Event> void post(T event) {
        List<Consumer<? extends Event>> eventListeners = listeners.get(event.getClass());
        if (eventListeners == null) return;
        
        // Рассылаем событие всем, кто на него подписался
        for (Consumer<? extends Event> listener : eventListeners) {
            ((Consumer<T>) listener).accept(event);
        }
    }
}
```

Если мы решим добавить в игру новую систему (например, ачивку *«Сруби 100 деревьев»*), **не придётся менять ни одной строчки кода** в обработчике ввода или в классе блоков. Достаточно создать `AchievementController`, подписать его на `BlockLeftClickEvent` и инкрементировать счётчик. Это и есть **слабая связанность (Loose Coupling)**.

## 4. Геймплейная логика: Процедурные мишени (Weak Spots)

Механика добычи ресурсов в Zenith — интерактивный процесс. Вместо монотонного закликивания стен, на блоках появляются процедурные слабые точки (**Weak Spots**), попадание по которым значительно ускоряет добычу.

```
      +--------------+          +--------------+
      |  (Грань куба)|          |  (Грань куба)|
      |              |          |      ()      | <-- Слабая точка (Weak Spot)
      |   X          |  ----->  |   X          |
      |   (Клик)     |          |   (Клик)     |
      +--------------+          +--------------+
```

### Как работает алгоритм Weak Spots:
1. **Рейкаст (Raycast):** Движок пускает луч из камеры вдоль взгляда игрока. Метод `raycast` находит точные 3D координаты пересечения луча с блоком и нормаль грани (какую именно сторону кубика мы бьем: бок, верх или низ).
2. **Проекция в UV:** Мы переводим трехмерную точку попадания на кубе в плоские двумерные координаты $(u, v)$ нашей грани (от 0.0 до 1.0).
3. **Генерация мишени:** При первом ударе на плоскости грани $(u, v)$ процедурно задается центр круглой мишени $(cx, cy)$ и радиус $R$.
4. **Смена стороны в реальном времени:** Если игрок продолжает зажимать кнопку удара, но оббегает блок вокруг, `MiningController` мгновенно фиксирует смену нормали грани и переносит мишень на новую грань прямо перед глазами игрока, сохраняя динамику боя!
5. **Проверка попадания:** Мы вычисляем расстояние от точки клика до центра мишени:
   $$\text{dist} = \sqrt{(u - cx)^2 + (v - cy)^2}$$
   Если $\text{dist} \le R$, игрок попал точно в яблочко!
   * Ему начисляется **тройной критический урон** по блоку.
   * Проигрывается сочный звонкий звук попадания.
   * `ParticleManager` выбивает целый сноп щепок (15 штук) вместо пары унылых опилок.
   * На блоке записывается глубокий физический шрам (зарубка) с помощью вектора `Vector4f` (интенсивность шрама в компоненте `w`).

#### 🌟 Эффект Wobble и Лечение блоков
При ударе по блоку создается временный прокси-меш. Шейдеры трясут его координаты по синусоиде, создавая упругую физическую отдачу (блок трясется от ударов!).
А если игрок бросил рубить дерево и ушел, через 5 секунд в мире запускается алгоритм лечения: здоровье блока постепенно восстанавливается, а зарубки и шрамы плавно теряют интенсивность `w`, затягиваясь на глазах за 2 секунды.

## 4.5 Input Handled-State: Ловушка для кликов (как укротить взбесившуюся мышку)

### В чём проблема?
Представьте ситуацию: вы сидите в кафе и листаете меню на планшете. Кликаете пальцем по экрану, чтобы выбрать пиццу. Но экран планшета оказался прозрачным, и клик не только нажал на кнопку «Заказать», но и проткнул чашку с кофе, стоявшую прямо за планшетом.

В геймдеве это называется **сквозным кликом (through-click)**. Вы закрываете инвентарь или кликаете по кнопке в меню паузы, а персонаж в этот же миг бьёт топором ценный верстак, на который наведена мышка. Чтобы этого не происходило, в **Zenith** внедрена система **Input Handled-State** (Состояние обработанного ввода).

### Метафора из жизни: Шлюзовая дверь
Это работает как классическая двойная дверь в шлюзе. Пока вы не закроете внешнюю дверь (UI-интерфейс), внутренняя дверь (игровой мир) физически заблокирована для любых манипуляций. Клик «поглощается» активным экраном и не летит дальше в глубины игрового пространства.

### Как это реализовано в Java
Посмотрим на схему диспетчеризации ввода в `InputManager`:

```java
// Упрощенный фрагмент логики диспетчеризации кликов в Zenith
public class InputManager {
    private Screen activeScreen; // Наше текущее открытое меню (инвентарь, настройки и т.д.)
    private WorldInteractionSystem worldInteraction; // Класс, отвечающий за разрушение блоков

    public void onMouseClick(double mouseX, double mouseY, int button) {
        // Шаг 1: Сначала спрашиваем активный UI-экран
        if (activeScreen != null) {
            // Передаем координаты мыши и кнопку в интерфейс
            boolean handled = activeScreen.handleMouseClick(mouseX, mouseY, button);
            
            // Если экран сказал: "Это по моей части, я обработал этот клик"
            if (handled) {
                // Прерываем выполнение метода! Клик успешно поглощен интерфейсом
                return; 
            }
        }
        
        // Шаг 2: И только если ни один UI-элемент не перехватил клик...
        // ...мы разрешаем игроку взаимодействовать с кубическим миром
        worldInteraction.performBlockRaycastAndInteract(button);
    }
}
```

## 4.6 Contextual Interaction: Интуитивный клик (швейцарский нож интерфейса)

### В чём проблема?
Представьте игру, где для каждого действия нужно открывать инвентарь или контекстное меню: «Открыть инвентарь -> Выбрать топор -> Выбрать пень -> Нажать "Положить бревно на пень"». Это утомляет и портит погружение. 

### Метафора из жизни: Умный сенсор на двери
Представьте, что вы подошли к двери с тяжелыми пакетами в руках. Дверь оборудована умным датчиком: она видит, что ваши руки заняты, и автоматически распахивается. Если вы подходите с пустыми руками — она ждет, пока вы нажмете на ручку. Система подстраивается под ваши физические условия.

В **Zenith** реализована система контекстного взаимодействия. Игра сама понимает, что вы хотите сделать, анализируя три фактора: на какой блок вы смотрите, что лежит у вас в руках, и в каком состоянии находится этот блок. Вся информация мгновенно выводится в аккуратный контекстный HUD (Heads-Up Display — информационную панель поверх экрана).

### Код выбора контекста взаимодействия в Java
Вся логика аккуратно разложена в `InteractionSystem`:

```java
public class InteractionSystem {
    public ContextualPrompt evaluateInteraction(Player player, BlockPos targetPos) {
        World world = player.getWorld();
        BlockState state = world.getBlockState(targetPos);
        ItemStack heldItem = player.getHeldItem();
        
        // Сценарий А: Мы смотрим на колоду для рубки (Stump)
        if (state.is(Blocks.STUMP)) {
            // Если в руках топор
            if (heldItem.is(Items.AXE)) {
                return new ContextualPrompt("ЛКМ: Рубить заготовку", PromptType.CRAFT);
            }
            // Если руки пусты, а на пне лежит готовое полено
            if (heldItem.isEmpty() && state.hasComponent(ContainerComponent.class)) {
                return new ContextualPrompt("ПКМ: Забрать полено", PromptType.INTERACT);
            }
            // Если в руке бревно, которое можно положить на колоду
            if (heldItem.is(Items.LOG)) {
                return new ContextualPrompt("ПКМ: Положить бревно на колоду", PromptType.PLACE);
            }
        }
        
        // Сценарий Б: Мы смотрим на куст диких ягод
        if (state.is(Blocks.BERRY_BUSH)) {
            if (state.get(BerryBushBlock.AGE) == 3) { // Куст созрел
                return new ContextualPrompt("ПКМ: Собрать ягоды", PromptType.HARVEST);
            }
        }

        // Если вариантов взаимодействия нет - возвращаем null (HUD чист)
        return null;
    }
}
```

## 5. In-World Crafting (Napping & Stump Crafting) без интерфейсов

Большинство RPG-игр при попытке создать предмет открывают огромное окно интерфейса на весь экран: с кнопочками, рецептами и кучей текста. Игра в этот момент замирает, а погружение в мир (Immersion) рушится.

В Zenith мы пошли по пути Hytale и реализовали **In-World Crafting** — систему крафта и обработки материалов прямо внутри игрового мира, без открытия окон GUI (Graphical User Interface — графического интерфейса пользователя).

### 5.1 Stump Crafting (Крафт на пне)
Чтобы сколотить инструмент, игрок находит физический блок пня (`StumpBlockEntity`).
1.  **Размещение ингредиентов:** Игрок кликает ПКМ (правой кнопкой мыши), держа в руках булыжник, палку или кремень.
2.  **Событийный ввод:** Событие `BlockRightClickEvent` ловится контроллером `InteractionController`.
3.  **In-World Container:** Предметы размещаются прямо на верхней грани пня как 3D-модели (благодаря компоненту `ContainerComponent`).
4.  **Удар топором:** Игрок бьет по пню топором (ЛКМ). Событие `BlockLeftClickEvent` передается в `StumpBlockEntity`.
5.  **Крафт:** Блок считывает рецепт из JSON, проигрывает анимацию удара, выбивает искры (частицы Shards) и заменяет лежащие ингредиенты на готовый каменный топор, который остается лежать на пне. Игрок может просто забрать его рукой!

### 5.2 Stone Napping (Обтёсывание кремня)
Для создания наконечника копья или каменного ножа игрок берет в руки два камня и бьет их друг об друга.
1.  **Сетка обтёсывания:** Перед глазами игрока прямо в мире проецируется плоская сетка 5x5 вокселей кремня.
2.  **Снятие лишнего:** Игрок кликает по ячейкам сетки, скалывая лишние кусочки камня. При каждом сколе вылетают каменные Shard-частицы и звучит звонкое «Дзынь!».
3.  **Consumption-логика событий:** Эти клики перехватываются `InteractionController` и «поглощаются» (`event.consume()`), блокируя стандартный взмах рукой или установку блоков из инвентаря.
4.  **Валидация рецепта:** Когда форма сколотого камня совпадает с паттерном из JSON-файла рецепта `recipes/napping_stone_knife.json`, процесс завершается: камень в руке превращается в острое каменное лезвие.

Благодаря Data-Driven рецептам в JSON и слабой связанности через `EventBus`, можно добавлять новые типы In-World обработки (например, ткачество или ковку на наковальне), не меняя ни одной строчки в физике или рендерере движка.

## 6. Готовимся к IT-собеседованию

### Вопрос 1: В чем суть паттерна EventBus (Шина событий)? Каковы плюсы и минусы его использования?
* **Ответ:** EventBus — это центральный канал обмена сообщениями между системами по принципу «Издатель-Подписчик». 
* **Плюсы:** Полная развязка систем (Loose Coupling). Класс ввода ничего не знает о звуках или частицах, новые фичи добавляются без переписывания старого кода.
* **Минусы:** Сложность отладки (Stack Trace прерывается, нельзя нажать `Ctrl+Click` на метод, чтобы увидеть вызывающий код). Опасность перегрузки GC (Garbage Collector — сборщика мусора), если создавать миллионы объектов-событий в секунду в игровом цикле.

### Вопрос 2: Как реализовать Zero-Alloc систему ввода для клавиатуры в Java?
* **Ответ:** Нужно отказаться от создания объектов-событий клавиш и не использовать коллекции вроде `Map<Integer, Boolean>` из-за оверхеда автобоксинга примитивов. Вместо этого состояние кнопок упаковывается в биты трех примитивных переменных `long` (`pressedMask`, `justPressed`, `justReleased`). Установка состояния кнопки делается побитовым ИЛИ: `pressedMask |= (1L << keyId)`, а опрос — побитовым И: `(pressedMask & (1L << keyId)) != 0L` за долю наносекунды без единой аллокации.

### Вопрос 3: Опишите алгоритм процедурных мишеней (Weak Spots) на блоках.
* **Ответ:** Мы пускаем луч взгляда в мир (Raycast), находим грань блока и переводим 3D-точку попадания в плоские координаты грани $(u, v)$ от 0 до 1. Процедурно задаем круглый диапазон мишени (центр и радиус). При кликах вычисляем евклидово расстояние от точки удара до центра мишени: если оно меньше радиуса — это крит (наносим тройной урон, высекаем зарубку и спавним сноп частиц).
