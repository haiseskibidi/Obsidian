## 1. Архитектура: CPU против GPU (Директор и Завод)

Перед тем как писать код рендеринга, давай разберемся, в чём физическая разница между центральным процессором (CPU) и видеокартой (GPU).

```
   ===================================    ===================================
   |             CPU CORE              |    |             GPU CORES             |
   |  +-----------+     +-----------+  |    |  +─+ +─+ +─+ +─+ +─+ +─+ +─+ +─+  |
   |  |   ALU 1   |     |   ALU 2   |  |    |  └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘  |
   |  +-----------+     +-----------+  |    |  +─+ +─+ +─+ +─+ +─+ +─+ +─+ +─+  |
   |  +-----------+     +-----------+  |    |  └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘  |
   |  |   Cache   |     | Control   |  |    |  +─+ +─+ +─+ +─+ +─+ +─+ +─+ +─+  |
   |  +-----------+     +-----------+  |    |  └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘  |
   |   Мало ядер, умные, быстрые       |    |   Тысячи ядер, простые, медленные |
   ===================================    ===================================
```

* **CPU (Центральный процессор) — это Умный Директор.**
  У него мало ядер (всего 8-16), но каждое ядро — это гений математики с огромным кэшем и сложнейшими блоками управления. Директор отлично справляется с ветвлениями (`if-else`), логикой игры, физикой и чтением файлов с диска. Но делать миллионы простейших расчетов одновременно — не его стихия.
* **GPU (Видеокарта) — это Огромный Автоматический Завод.**
  Тут работают тысячи крошечных, очень простых рабочих-ядер. Они туповатые: не умеют быстро делать сложные логические переходы. Но они умеют делать одну простейшую операцию (например, умножить число А на число Б) одновременно над миллионами пикселей.

3D-графика — это миллиарды пикселей и вершин, к которым нужно применить абсолютно одинаковые формулы. Идеальная задача для нашего автоматического завода!

## 2. Стадии графического конвейера (Rendering Pipeline) на пальцах

Когда ты хочешь нарисовать на экране хотя бы один треугольник, данные проходят длинный путь через конвейер. Большинство стадий конвейера аппаратно зашиты в видеокарту, но две главные стадии мы пишем сами на языке **GLSL** (OpenGL Shading Language — языке программирования шейдеров) — они называются **шейдерами (Shaders)**.

```
  +--------------+     +--------------+     +--------------+
  | Vertex Spec  |---->| Vertex Sh.   |---->| Rasterization|
  | (VAO/VBO)    |     | (Програм.)   |     | (Аппаратная) |
  +--------------+     +--------------+     +──────┬───────+
                                                   │
  +--------------+     +--------------+            │
  | Framebuffer  |<----| Per-Sample   |<-----------+
  | (Экран/VRAM) |     | (Depth/Blend)|
  +--------------+     +--------------+
```

Давай разберем конвейер по шагам:

1. **Vertex Specification (Сбор вершин):** Мы подготавливаем координаты нашего треугольника в Java и загружаем их в видеопамять (VRAM).
2. **Vertex Shader (Вершинный шейдер) — `ПИШЕМ САМИ`:** 
   Этот робот запускается отдельно для каждой вершины треугольника. Его задача — перевести трехмерные координаты из игрового мира $(X,Y,Z)$ в плоские координаты экрана (от -1.0 до 1.0) с учетом того, куда смотрит камера и какой у игрока угол обзора (FOV — Field of View, поле зрения). Результат он записывает в системную переменную `gl_Position`.
3. **Rasterization (Растеризация) — `АППАРАТНАЯ СТАДИЯ`:**
   Видеокарта берет треугольник и накладывает его на сетку пикселей монитора. Она определяет, какие именно пиксели покрывает треугольник, и превращает его в набор **фрагментов** (кандидатов в пиксели).
4. **Fragment Shader (Фрагментный шейдер) — `ПИШЕМ САМИ`:**
   Этот робот-маляр запускается для каждого пикселя, который покрыл наш треугольник. Его единственная задача — вернуть цвет пикселя (красный, синий или цвет из текстуры). Результат записывается в переменную `FragColor`.
5. **Per-Sample Operations (Тесты и смешивание) — `АППАРАТНАЯ СТАДИЯ`:**
   Видеокарта проверяет, виден ли этот пиксель игроку. 
   * *Depth Test (Тест глубины):* Если перед нашим треугольником уже нарисована каменная стена ближе к камере, этот пиксель просто выбрасывается, чтобы не тратить силы.
   * *Blending (Смешивание):* Если объект прозрачный (например, стекло), его цвет смешивается с цветом фона.
6. **Framebuffer:** Финальная картинка выводится на экран.

## 2.5 Modular Rendering Pipeline: Оркестр без дирижёра-монолита

### В чём проблема?
В ранних прототипах класс `Renderer.java` напоминал безумного швейцарского армейского ножа. Он умел делать абсолютно всё: инициализировать OpenGL, считать матрицы камеры, грузить меши, рисовать небо, чанки, игроков, частицы, обводку блоков и меню паузы. Код превратился в непролазные джунгли на 10 000 строк. Любое изменение в отрисовке травы ломало интерфейс.

### Метафора из жизни: Кухня мишленовского ресторана
В **Zenith** мы провели великую реформу и разделили этот монолит на профессиональную команду:
1. **RenderPipeline**: Это наш шеф-повар. Он сам не чистит картошку и не моет посуду. Он лишь координирует фазы кадра (оркестратор жизненного цикла).
2. **RenderContext**: Это разделочный стол и кухонный инвентарь. Здесь лежат все специи (матрицы JOML — математической библиотеки для Java, буферы данных) в готовом виде. Никаких аллокаций памяти в процессе готовки!
3. **MeshRegistry**: Кладовая заготовок. Хранит готовые 3D-модели (меши) для блоков и предметов.
4. **ChunkRenderSystem**, **EntityRenderSystem**, **OverlayRenderSystem**: Специализированные повара горячего, холодного и кондитерского цехов.

### Схема жизненного цикла кадра в Java

```java
public class RenderPipeline {
    private final RenderContext context;
    private final ChunkRenderSystem chunkRenderer;
    private final EntityRenderSystem entityRenderer;
    private final OverlayRenderSystem overlayRenderer;

    public void render(float deltaTime) {
        // Шаг 1: Подготовка сцены (Очищаем экран, заливаем буфер глубины, обновляем UBO — Uniform Buffer Object, буфер глобальных констант)
        context.prepareFrame();

        // Шаг 2: Отрисовка ландшафта (Повар горячего цеха берется за дело)
        // Рисуем все непрозрачные и прозрачные блоки мира
        chunkRenderer.render(context);

        // Шаг 3: Отрисовка динамических объектов (Повар холодного цеха)
        // Мобы, падающие предметы, руки игрока с инструментами
        entityRenderer.render(context);

        // Шаг 4: Наложение эффектов (Декоратор украшает блюдо)
        // Обводка блока под прицелом, трещины разрушения, шрамы на блоках
        overlayRenderer.render(context);

        // Шаг 5: Завершение кадра (Уборка на кухне)
        // Очищаем Zero-Allocation пулы векторов и матриц для следующего кадра
        context.endFrame();
    }
}
```

## 3. Буферы данных в OpenGL (VAO, VBO, EBO) — Вся правда о контейнерах

Чтобы передать данные о блоках и руках персонажа из Java на видеокарту, OpenGL использует три основных типа буферов. Давай разберем их суть с помощью нашей метафоры о Заводе.

### 3.1 VBO (Vertex Buffer Object) — Огромная Фура с кирпичами
**VBO** — это просто длинный плоский массив байт в видеопамяти GPU. 
Представь, что мы отправили на склад Завода (VRAM) гигантскую фуру, доверху забитую кирпичами. Все кирпичи лежат в один ряд на полу фуры:
`[Координаты 1, Текстура 1, Координаты 2, Текстура 2...]`
Для Завода это просто сырые байты. Он понятия не имеет, где тут заканчивается координата и начинается цвет. Ему нужен описатель груза.

### 3.2 VAO (Vertex Array Object) — Таможенная декларация
**VAO** не хранит координаты сам. Он хранит настройки — описание того, что лежит в фуре VBO, чтобы рабочие завода могли правильно достать детали:
* *«Первые 3 ящика (12 байт) — это координаты вершин (float x, y, z)»*
* *«Следующие 2 ящика (8 байт) — это текстурные координаты (float u, v)»*
* *«Шаг между деталями (Stride) — 20 байт»*

Вот как это описывается в коде Java:
```java
GL30.glBindVertexArray(vaoId); // 1. Активируем декларацию VAO
GL15.glBindBuffer(GL15.GL_ARRAY_BUFFER, vboId); // 2. Связываем с фурой VBO

// 3. Описываем атрибут 0: Координаты (3 float, шаг 32 байта, смещение 0)
GL20.glVertexAttribPointer(0, 3, GL11.GL_FLOAT, false, 32, 0);
GL20.glEnableVertexAttribArray(0);

// 4. Описываем атрибут 1: Текстурные координаты (2 float, шаг 32 байта, смещение 12 байт)
GL20.glVertexAttribPointer(1, 2, GL11.GL_FLOAT, false, 32, 12);
GL20.glEnableVertexAttribArray(1);
```

### 3.3 EBO / IBO (Element / Index Buffer Object — буфер индексов) — Инструкция по сборке
Представь, что мы рисуем воксельный куб. У куба 6 граней. Каждая грань состоит из 2 треугольников. Итого 12 треугольников. У каждого треугольника по 3 вершины — всего нужно описать **36 вершин**. 

Но у куба физически всего **8 углов**! Зачем нам дублировать координаты одних и тех же углов по 5 раз, забивая память видеокарты?

Мы используем **EBO (Буфер индексов)**:
1. В **VBO** записываем координаты всего 8 уникальных углов куба (под индексами от 0 до 7).
2. В **EBO** записываем простую инструкцию-схему для сборщиков завода: 
   *«Соберите первый треугольник из углов №0, №1 и №2»*
   *«Второй треугольник соберите из углов №2, №3 и №0»*
   `EBO = [0, 1, 2, 2, 3, 0, ...]`

Это сокращает объем передаваемых на видеокарту данных в **3-4 раза**!

## 4. Проблема Draw Calls Bottleneck: почему игра тормозит?

Это величайший кошмар всех начинающих разработчиков графики.

Когда твой процессор (CPU) хочет нарисовать объект, он вызывает команду:
`GL11.glDrawElements(GL11.GL_TRIANGLES, count, GL11.GL_UNSIGNED_INT, 0);`
Этот вызов называется **Draw Call (Команда отрисовки)**.

Что происходит в этот момент внутри компьютера?
1. Директор (CPU) отрывается от дел, проверяет состояние OpenGL.
2. Конвертирует Java-команду в вызов драйвера видеокарты.
3. Драйвер передает команду операционной системе, та отправляет её по шине PCIe (высокоскоростной шине передачи данных) на видеокарту.

А теперь представь: в твоем воксельном мире отрисовывается **5 000 отдельных блоков листвы и травы**. И ты в цикле делаешь 5 000 вызовов `glDrawElements()`. 

Директор (CPU) будет судорожно крутить диск старого телефона, пытаясь дозвониться до завода 5 000 раз за кадр. Телефонная линия PCIe перегружена. CPU загружен на 100% только обслуживанием звонков. А огромный завод (GPU) выполняет сборку блока за наносекунду и **простаивает 99% времени**, скучая в ожидании следующего звонка. 

В итоге FPS падает до 15, видеокарта холодная (загрузка 5%), а процессор кипит!

### Ужас переключения состояний (Pipeline Stall)
Еще хуже, если между звонками директор меняет настройки:
* *«А теперь красим в красный!»* (`glBindTexture` для текстуры А) -> *«Рисуем!»* -> *«А теперь красим в синий!»* (`glBindTexture` для текстуры Б) -> *«Рисуем!»*

Каждый такой вызов заставляет конвейер видеокарты полностью остановиться (**Pipeline Stall**). Рабочие завода бросают инструменты, гасят печи, ждут, пока остынет старый клей (очистятся текстурные кэши), перенастраивают станки под новую текстуру и только потом продолжают работу. Микросекундные простои складываются в гигантские лаги.

> [!IMPORTANT]
> **Золотое правило Lead Developer'а:** Минимизируй количество Draw Calls и переключений состояний! Рисуй тысячи объектов за один единственный звонок, упаковывая их в общие буферы (Батчинг).

## 5. Разбираем простейшие Шейдеры (GLSL)

Шейдеры пишутся на языке GLSL (очень похож на Си). Давай разберем эталонную пару шейдеров, которые рисуют текстурированный объект с матрицами.

### Вершинный шейдер (`vertex.glsl`)
```glsl
#version 330 core

// 1. Принимаем атрибуты вершины из фуры VBO по схеме из декларации VAO
layout(location = 0) in vec3 aPos;       // Координаты (x, y, z)
layout(location = 1) in vec2 aTexCoords; // Текстурные координаты (u, v)

// 2. Выходной стикер, который полетит дальше во фрагментный шейдер
out vec2 TexCoords;

// 3. Глобальные константы (настройки камеры и положения объекта)
uniform mat4 uProjectionView; // Матрица камеры (FOV + положение в мире)
uniform mat4 uModel;          // Матрица положения объекта (куда мы его поставили)

void main() {
    // Умножаем матрицы на координаты вершины. 
    // Вектор aPos переводится из локальных координат в мировые, а затем на экран!
    gl_Position = uProjectionView * uModel * vec4(aPos, 1.0);
    
    // Наклеиваем текстурные координаты на выходной стикер
    TexCoords = aTexCoords;
}
```

### Фрагментный шейдер (`fragment.glsl`)
```glsl
#version 330 core

// 1. Принимаем стикер из вершинного шейдера
in vec2 TexCoords;

// 2. Финальный выходной цвет пикселя на экране
out vec4 FragColor;

// 3. Текстура, загруженная в видеопамять
uniform sampler2D uTexture;

void main() {
    // Робот-маляр берет цвет из текстуры по координатам TexCoords
    vec4 texColor = texture(uTexture, TexCoords);
    
    // Если пиксель прозрачный (например, дырка в текстуре листвы) — 
    // мгновенно выбрасываем этот пиксель из рендеринга!
    if (texColor.a < 0.1) {
        discard; 
    }
    
    // Красим пиксель в цвет текстуры
    FragColor = texColor;
}
```

> [!NOTE]
> **Как работает интерполяция?**
> Обрати внимание: Вершинный шейдер запускается всего 3 раза (для 3 углов треугольника). А фрагментный шейдер запускается миллионы раз (для каждого пикселя внутри треугольника). Видеокарта автоматически **плавно интерполирует (сглаживает)** координаты `TexCoords` между углами треугольника, чтобы фрагментный шейдер для каждого пикселя получил его точную индивидуальную координату на текстуре!

## 5.2 Unified Coloring: Элегантная покраска без лишней суеты

### В чём проблема?
Представь, что ты красишь забор. В игре сотни биомов: зелень джунглей, жухлая трава саванны, багряная листва осеннего леса. Если для каждого биома создавать отдельную цветную текстуру, видеопамять (VRAM) закончится ещё на этапе загрузки. А если пересчитывать цвета на CPU и пересоздавать текстуры на лету — получим слайд-шоу.

### Метафора из жизни: Трафаретное граффити
Решение в **Zenith** гениально и просто, как детская раскраска по номерам. Базовые текстуры травы и листвы мы храним на диске абсолютно **серыми** (монохромными). Это наш холст.
А в шейдер мы передаем специальный "трафарет" — четвертую текстурную координату (`OverlayLayer`).
*   Если значение `OverlayLayer` равно `0.0` — это защищенная зона (например, сухая веточка или цветок на кусте). Красить её нельзя!
*   Если `OverlayLayer` равно `1.0` — это зона для покраски.
*   Центральный класс `ColorProvider` выступает в роли "колориста", который точно знает текущий цвет биома. Шейдер берёт серый пиксель, накладывает трафарет и распыляет цвет биома поверх, сохраняя все тени и текстурные прожилки!

### GLSL Шейдер «на пальцах» (Кусочек `fragment.glsl`)
Давай посмотрим, как это происходит прямо в графическом чипе:

```glsl
#version 330 core

// Входные данные от вершинного шейдера
in vec2 vTexCoord;        // Обычные текстурные координаты (U, V)
in float vOverlayLayer;   // Наш трафарет: 1.0 = красим в цвет биома, 0.0 = не трогаем

uniform sampler2D uTexture; // Наша базовая серая текстура
uniform vec3 uBiomeColor;   // Сочный цвет биома, полученный из ColorProvider

out vec4 FragColor;         // Итоговый цвет пикселя на экране

void main() {
    // 1. Достаем исходный цвет пикселя из серой текстуры (RGB + прозрачность)
    vec4 baseColor = texture(uTexture, vTexCoord);
    
    // 2. Магическое смешивание (mix)! 
    // Если vOverlayLayer = 0.0, mix вернет vec3(1.0) (белый цвет, при умножении ничего не меняет).
    // Если vOverlayLayer = 1.0, mix вернет чистый uBiomeColor.
    // Если значение промежуточное — плавно смешает их.
    vec3 tint = mix(vec3(1.0), uBiomeColor, vOverlayLayer);
    
    // 3. Умножаем исходную серость на наш тинт. 
    // Серая текстура работает как карта яркости: темные места останутся темными, светлые — светлыми!
    vec3 finalColor = baseColor.rgb * tint;
    
    // 4. Выплескиваем пиксель на экран, не забывая сохранить прозрачность (альфа-канал)
    FragColor = vec4(finalColor, baseColor.a);
}
```

## 5.3 SDF-фигуры (Procedural SDF Rendering): Идеальные круги без пикселей

### В чём проблема?
Как нарисовать идеально четкую круговую шкалу, кольцо радара или сонар на миникарте? Если использовать обычные картинки (текстуры), то при приближении или изменении разрешения экрана они превратятся в размытое пиксельное месиво. 

### Метафора из жизни: Колышек с веревкой
В **Zenith** используется **Procedural SDF Rendering** (Signed Distance Fields — знакопеременные поля расстояний). Видеокарта рисует фигуры не по готовым картинкам, а по чистым математическим формулам прямо в шейдере `minimap_fragment.glsl`. Результат — бесконечная, векторная четкость при любом масштабе!

Представь, что вам нужно нарисовать ровный круг на асфальте. Вы можете принести огромный картонный трафарет (это текстура), но если асфальт неровный или трафарет помялся, круг выйдет кривым. Альтернатива — взять колышек, привязать веревку определенной длины и прочертить идеальный круг мелом. SDF — это и есть та самая «математическая веревка» внутри вашей видеокарты.

### GLSL Код «на пальцах»

Кусочек шейдера GLSL, который вычисляет и рисует идеально четкое кольцо сонара:

```glsl
/**
 * Процедурно генерирует идеальное кольцо сглаженным образом.
 * p - координаты пикселя на экране (от -1.0 до 1.0 относительно центра).
 * radius - радиус кольца.
 * thickness - толщина линии кольца.
 */
float drawSDFRing(vec2 p, float radius, float thickness) {
    // 1. Находим расстояние от центра (0,0) до текущей точки p
    float d = length(p);
    
    // 2. Вычисляем, насколько этот пиксель удален от идеальной окружности
    float distToCircle = abs(d - radius);
    
    // 3. Функция smoothstep делает края линии невероятно гладкими (сглаживание/antialiasing),
    // превращая математическое расстояние в мягкую маску прозрачности.
    return smoothstep(thickness, 0.0, distToCircle);
}
```

## 5.4 Dynamic VoxelShape Highlight: Живой неон контуров

### В чём проблема?
Когда вы наводите взгляд на блок ступенек, игра должна красиво обвести его контур. Но обычная рамка обвела бы каждый внутренний кубик ступеньки отдельно, превратив её в мешанину светящихся линий. 

### Метафора из жизни: Неоновая лента на ступенях
В **Zenith** алгоритм **Algorithmic Wireframes** динамически анализирует форму блока, «сшивает» соприкасающиеся грани соседних AABB (ось-выровненных ограничивающих параллелепипедов) и удаляет все внутренние линии. На выходе получается идеальный цельный контур. 
Важная деталь: этот контур синхронизирован с физической анимацией удара (Impact Wobble). Линии контура передаются в вершинный шейдер `vertex.glsl` с флагом `uIsProxy`. При ударе по блоку рамка сжимается, дрожит и вибрирует синхронно с физической моделью.

Представь, что вы хотите обернуть неоновой лентой ступенчатую пирамиду. Вы не будете обматывать каждую ступеньку отдельно изнутри — вы просто натяните ленту по внешним ребрам конструкции. А если пирамида начнет трястись от землетрясения, натянутая лента будет дрожать вместе с ней как единое целое.

### Как устроено сшивание ребер рамки на Java:

```java
/**
 * Генерирует очищенную сетку контура блока, удаляя невидимые ребра на стыках.
 */
public List<Line> generateStitchedHighlight(VoxelShape shape) {
    List<Line> resultLines = new ArrayList<>();
    List<AABB> boxes = shape.getBoxes();
    
    // Проходим по каждой коробочке внутри сложной формы блока
    for (AABB box : boxes) {
        // Берем все 12 ребер этой коробочки
        for (Line edge : box.getEdges()) {
            boolean isInsideOtherBox = false;
            
            // Проверяем, не утопает ли это ребро внутри другой соседней коробочки
            for (AABB other : boxes) {
                if (other == box) continue;
                
                // Если середина ребра находится внутри соседнего AABB - ребро скрыто от глаз!
                if (other.containsPoint(edge.getMidpoint())) {
                    isInsideOtherBox = true;
                    break;
                }
            }
            
            // Если ребро внешнее и свободное - добавляем его в список на отрисовку!
            if (!isInsideOtherBox) {
                resultLines.add(edge);
            }
        }
    }
    return resultLines;
}
```

## 5.5 AAA Graphics & Post-Stack: Кинематографический глянец и glDepthRange

### В чём проблема?
Многие инди-игры грешат противным подергиванием картинки при движении камеры. Физика в мире считает координаты с фиксированной частотой (например, 20 тиков в секунду или 170 тиков в секунду), а монитор работает на частоте 144 Гц или 240 Гц. Если обновлять картинку строго по физическим тикам — вы получите рваное слайд-шоу. 

Кроме того, при наложении спецэффектов (FXAA — быстрое сглаживание пикселей, виньетка, туман) частые проходы по текстурам забивают видеопамять. А руки и инструменты персонажа (кирка, факел) проваливаются сквозь текстуры стен, когда вы стоите к ним вплотную.

### Решение: Разделение потоков и glDepthRange
В **Zenith** эта проблема решена на уровне **AAA-архитектуры интерполяции**: физика и рендер полностью разделены. Физический движок считает шаги в своем темпе, а рендерер вычисляет промежуточное положение камеры и объектов между кадрами с помощью коэффициента `alpha`. А сглаживание и анизотропная фильтрация сглаживают пиксельные ребра блоков, делая картинку невероятно чистой.

Движок Zenith собирает все эффекты пост-обработки в единый проход рендеринга (**Single Pass**). Это экономит кучу ресурсов видеокарты. 

А чтобы руки и инструменты персонажа не проваливались сквозь текстуры стен, используется магия `glDepthRange`. Этот метод сжимает диапазон глубин для рук, заставляя видеокарту думать, что они всегда находятся на самом переднем плане, прямо перед глазами, даже если физически они пересекают стену!

### Метафора из жизни: Театральная сцена
Представьте себе мультфильм. Художник-аниматор нарисовал всего 10 ключевых кадров бега персонажа (физические тики). Но хитрый проектор прогоняет пленку через специальный процессор, который плавно дорисовывает промежуточные фазы движения между этими кадрами, превращая дерганый мультик в плавное кино.
А трюк с руками — это как театральная сцена, где актера выдвигают на специальный подиум ближе к зрителям, чтобы декорации заднего плана его случайно не заслонили.

### Код пост-процессинг шейдера, объединяющий эффекты в один проход:

```glsl
uniform sampler2D uScreenTexture; // Наша исходная картинка игры
uniform sampler2D uDepthTexture;  // Текстура глубины кадра
in vec2 uv;                       // Координаты текущего пикселя (от 0.0 до 1.0)
out vec4 FragColor;               // Финальный цвет пикселя на мониторе

void main() {
    // 1. Читаем базовый цвет пикселя
    vec3 color = texture(uScreenTexture, uv).rgb;
    
    // 2. Вычисляем виньетку (затемнение по углам экрана)
    vec2 distFromCenter = uv - vec2(0.5);
    float vignette = 1.0 - dot(distFromCenter, distFromCenter) * 0.4;
    color *= vignette; // Применяем легкий голливудский драматизм
    
    // 3. Читаем глубину из текстуры глубины
    float depth = texture(uDepthTexture, uv).r;
    
    // 4. Накладываем атмосферный туман
    float fogFactor = calculateFogFactor(depth);
    vec3 fogColor = vec3(0.5, 0.6, 0.7); // Приятный сизый туман
    color = mix(color, fogColor, fogFactor);
    
    // Выводим сочный, сглаженный и обработанный кадр!
    FragColor = vec4(color, 1.0);
}
```

## 6. Готовимся к IT-собеседованию

### Вопрос 1: В чем разница между VAO и VBO? Зачем нужны оба буфера?
* **Ответ «для чайников»:** VBO — это физический склад (сырой одномерный массив байт в видеопамяти), где лежат координаты вершин. VAO — это таможенная декларация (описатель формата). Он объясняет видеокарте, как правильно читать сырые байты из VBO: где лежат координаты, где текстуры, с каким шагом и смещением. Без VAO видеокарта не поймет, что за каша из байт лежит в VBO.

### Вопрос 2: Что такое Draw Call и почему большое количество Draw Calls «убивает» FPS?
* **Ответ «для чайников»:** Draw Call — это системная команда отрисовки группы полигонов. Если объектов на сцене много и мы рисуем их по очереди в цикле, процессор тратит кучу времени на медленные звонки драйверу видеокарты по шине PCIe. Процессор перегружается звонками, а видеокарта простаивает без работы, из-за чего FPS падает.

### Вопрос 3: Как работает тест глубины (Depth Test) и почему порядок отрисовки объектов критически важен?
* **Ответ «для чайников»:** Тест глубины проверяет Z-координату пикселя: если новый пиксель находится дальше от камеры, чем уже нарисованный в этой точке экрана объект, видеокарта выбрасывает его.
* **Оптимизация:**
  1. *Непрозрачные объекты* нужно рисовать **от ближних к дальним** (Front-to-Back). Видеокарта нарисует ближнюю стену, а дальние горы мгновенно отбросит по тесту глубины, даже не запуская для них тяжелые фрагментные шейдеры (это экономит кучу сил видеокарты).
  2. *Прозрачные объекты* нужно рисовать **в самом конце и строго от дальних к ближним** (Back-to-Front), чтобы цвета стёкол правильно смешивались с фоном. При этом запись в Z-буфер (буфер глубины) временно выключается.
