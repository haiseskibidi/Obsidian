# Глава 1. Управление памятью JVM & Стратегия Zero-Allocation

Это первая глава практического руководства — и, пожалуй, самая фундаментальная.

Здесь не будет сухих академических определений. Мы разберём темы управления памятью «на пальцах», с живыми примерами и детальным разбором каждой строчки кода.

Наша цель — понять, почему стандартный Java-код, который отлично и быстро работает в веб-сервисах или консольных утилитах, в играх реального времени с треском проваливается, вызывая жуткие микрофризы. И, конечно же, мы научимся писать код по стратегии **Zero-Allocation (Нулевое выделение памяти)**, как в настоящих игровых движках вроде **Zenith**.

## 1. Стек и Куча (Stack vs Heap) на пальцах

Давай представим память компьютера, выделенную под нашу Java-программу, в виде уютной квартиры, где живут два совершенно разных порядка: **Стек (Stack)** и **Куча (Heap)**.

```
       ==========================================================
       |                 Оперативная память JVM                 |
       +---------------------------+----------------------------+
       |     СТЕК (Stack)          |       КУЧА (Heap)          |
       |  (Упорядоченная кухня)    |   (Детская с игрушками)    |
       |                           |                            |
       |  +---------------------+  |  +----------------------+  |
       |  | Стопка тарелок:     |  |  | [Новая игрушка]      |  |
       |  |   float dt = 0.016f |  |  | Vector3f position    |  |
       |  |   Ссылка ref  ------+--+->|   x = 10, y = 5...   |  |
       |  +---------------------+  |  +----------------------+  |
       |  Супер-быстро, чисто      |  Медленно, убирает мама    |
       ==========================================================
```

### 1.1 Стек — это стопка тарелок на кухне
Стек (Stack) — это личная память каждого отдельного потока (Thread). Он работает по принципу **LIFO (Last In, First Out)** — «последним пришел, первым ушел». 

Представь стопку чистых тарелок. Ты кладешь новую тарелку на самый верх и берешь тарелку тоже строго сверху.
Когда программа вызывает какой-то метод (например, `update()`), JVM (виртуальная машина Java) берет чистую тарелку — **Кадр стека (Stack Frame)** — и кладет наверх стопки. В эту тарелку складываются:
1. Локальные примитивные переменные (числа типа `int = 5`, `float dt = 0.016f`, флажки `boolean`).
2. **Стикеры со стрелочками** (ссылки на объекты в куче).

Как только метод заканчивает работу, JVM просто берёт эту верхнюю тарелку со стола и выбрасывает. Мгновенно, за доли наносекунды. Память очистилась сама, без всяких уборщиков и усилий.

### 1.2 Куча — это детская комната с разбросанными игрушками
Куча (Heap) — это общая память для всей программы. И тут царит совсем другая атмосфера.

Представь, что в квартире играет ребенок (наш игровой код). Он бегает по комнате и каждые полсекунды кричит: 
* *«Хочу новую машинку!»* (код пишет: `new Vector3f()`)
* *«Хочу нового мишку!»* (код пишет: `new BlockPos()`)

Каждый раз, когда в коде пишется ключевое слово `new`, JVM выделяет память в **Куче**. Но наш ребенок капризный: он берет игрушку, играет с ней долю секунды, бросает прямо на пол и бежит создавать новую. Через пару минут комната завалена тысячами брошенных «мертвых» игрушек. Это и есть **мусор в памяти**.

Куча не умеет очищаться сама при выходе из метода. Если бросить игрушку на пол, она так и будет там валяться.

## 2. Ужас Сборщика Мусора (Garbage Collector) в играх

И вот, когда свободное место в комнате (куче) полностью заканчивается, в детскую заходит строгая мама — **Сборщик Мусора (Garbage Collector / GC)**. 

Ей нужно навести порядок. Но как это сделать, если ребенок продолжает носиться по комнате и швырять игрушки? Правильно, никак. Поэтому мама громко командует: **«А ну-ка, все замерли на месте!»**.

В программировании это называется **Stop-the-World (STW)**. 

JVM полностью останавливает выполнение всех игровых потоков (включая поток рендеринга и обсчета физики). Мама-GC берет веник и начинает кропотливо искать, какие игрушки еще нужны ребенку (живые ссылки из стека), а какие можно выкинуть в мусорный бак (мертвые объекты). 

* **Почему это катастрофа для геймдева?**
  Если игра идет на частоте 60 FPS (кадров в секунду), то один кадр должен рендериться за **16.6 миллисекунд** (`1 секунда / 60`). Если у нас 144 FPS (монитор 144 Гц), то время кадра сжимается до безумных **6.9 миллисекунд**!
  Если сборщик мусора остановит игру хотя бы на **10-15 миллисекунд** для уборки в куче, поток рендеринга не успеет вовремя отдать кадр видеокарте. Игрок мгновенно увидит микрофриз (дерганье экрана). FPS резко упадет, а плавность управления («Game Feel») превратится в тыкву.

## 3. Анатомия преступления: разбираем плохой код

Давай досконально, по косточкам разберем код, который и вызвал твои трудности. Это классический пример того, как писать **НЕ надо**, если ты хочешь выжать из движка максимум производительности.

```java
// Вызывается 200 раз в секунду в игровом цикле!
public void update(float deltaTime) {
    for (Entity entity : entities) {
        Vector3f deltaPos = new Vector3f(entity.velocity).mul(deltaTime); // МИНА №1!
        entity.position.add(deltaPos);
        
        BlockPos currentBlock = new BlockPos(entity.position); // МИНА №2!
        // ...
    }
}
```

Давай разберем каждое слово:
* `update(float deltaTime)` — это метод обновления игровой физики. В современных динамичных играх его гоняют очень часто. В Zenith он крутится **200 раз в секунду** (частота 200 Гц), чтобы коллизии и прыжки рассчитывались с хирургической точностью.
* `float deltaTime` — время, прошедшее с предыдущего кадра. Так как компьютеры работают с разной скоростью, мы не можем просто прибавлять фиксированное число к координатам. Мы берем скорость объекта и умножаем на `deltaTime`. Если игра выдает 60 кадров в секунду, то `deltaTime` равен примерно `0.016` секунды.
* `for (Entity entity : entities)` — перебираем всех существ в нашем игровом мире (игрока, зомби, летящие стрелы, частицы). Допустим, на карте сейчас скромно бегает **100 сущностей**.

### А теперь смотри, где зарыта бомба замедленного действия:

#### 💣 Мина №1: `Vector3f deltaPos = new Vector3f(entity.velocity).mul(deltaTime);`
Мы хотим узнать, на какое расстояние сдвинулся зомби за этот кадр. Мы берем вектор его скорости `entity.velocity` (например, `[x=0, y=0, z=3]`), делаем его копию через `new Vector3f(...)` и умножаем на время. 
Но мы использовали `new`! Это значит, мы выделили память в куче.

Давай посчитаем математику мусора:
`100 сущностей * 200 вызовов в секунду = 20 000 объектов в секунду!`
Всего одна невинная строчка создает **двадцать тысяч** временных векторов в секунду! А за минуту игры это **1.2 миллиона** ненужных объектов! И все они отправляются на пол детской комнаты.

#### 💣 Мина №2: `BlockPos currentBlock = new BlockPos(entity.position);`
Мы хотим узнать, в каком воксельном блоке сейчас стоит сущность. Мы берем ее трехмерные координаты типа `float` (`[12.4f, 64.0f, -8.9f]`) и округляем до целых чисел, создавая объект `BlockPos` (`[x=12, y=64, z=-9]`).
И снова пишем `new`! Получаем еще **20 000 объектов в секунду**!

Суммарно этот короткий цикл генерирует гигабайты мусора из ничего. Через 5-10 секунд куча переполняется, прибегает мама-GC с веником, кричит «Stop the World!», игра замирает на 20 мс, и зомби телепортируется на метр вперед из-за лага. Игрок в ярости.

## 4. Спасение: Стратегия Zero-Allocation в Zenith

Как нам спасти ситуацию и успокоить сборщик мусора? Использовать стратегию **Zero-Allocation (Нулевое выделение памяти)**!
Главное правило бойцовского клуба Zero-Alloc: **Внутри игрового цикла (Update/Render) ключевое слово `new` находится под строжайшим запретом!**

Как же тогда считать физику и координаты, если нельзя создавать новые объекты? Давай перепишем наш плохой код правильно!

```java
// 1. Создаем ОДИН вспомогательный вектор заранее при запуске игры.
// Он лежит в куче навсегда, и мы его никогда не выбрасываем.
private final Vector3f tempDelta = new Vector3f();

// 2. Вместо создания объектов BlockPos, используем одно переиспользуемое "изменяемое" поле
private final BlockPos.Mutable tempBlockPos = new BlockPos.Mutable();

public void update(float deltaTime) {
    for (Entity entity : entities) {
        // Забываем про new! 
        // Мы берем нашу готовую многоразовую "коробку" tempDelta,
        // наливаем в неё значения скорости сущности и умножаем на время.
        tempDelta.set(entity.velocity).mul(deltaTime); // 0 аллокаций!
        
        // Прибавляем результат к позиции сущности.
        entity.position.add(tempDelta); // 0 аллокаций!
        
        // Переиспользуем mutable-объект, просто меняя его внутренние x, y, z
        tempBlockPos.set(entity.position.x, entity.position.y, entity.position.z); // 0 аллокаций!
        
        // Теперь мы можем безопасно работать с координатами блока!
        int blockId = world.getBlockType(tempBlockPos);
    }
}
```

Результат: абсолютно те же расчёты, но **ни одного нового объекта** в куче. Garbage Collector спит, комната чистая, FPS стабильно держится на 144 Гц.

## 5. Пул Объектов (Object Pooling): когда нужно много и сразу

С одной переменной `tempDelta` всё понятно. А что делать, если в разных методах рендеринга нужно одновременно рассчитывать десятки матриц трансформации и векторов? Заводить 100 полей `tempMatrix1`, `tempMatrix2` — не вариант.

Конечно нет. Для этого умные люди придумали паттерн **Object Pool (Пул объектов)**.

Представь городскую систему проката самокатов. Вместо того чтобы покупать новый самокат каждый раз, когда тебе нужно доехать до магазина, а потом бросать его в реку (`new`), ты берешь самокат со стоянки, доезжаешь до места, а потом возвращаешь его обратно на парковку.

Вот как устроен простейший круговой пул объектов в движке **Zenith**:

```java
// com.za.zenith.engine.graphics.RenderContext.java
public class RenderContext {
    // Наша "стоянка" на 128 матриц и 256 векторов
    private final Matrix4f[] matrixPool = new Matrix4f[128];
    private int matrixIndex = 0;

    public RenderContext() {
        // Заполняем пул ОДИН раз при старте игры
        for (int i = 0; i < matrixPool.length; i++) {
            matrixPool[i] = new Matrix4f();
        }
    }

    // Выдать свободную матрицу из пула
    public Matrix4f getMatrix() {
        // Если дошли до конца пула — возвращаемся в начало (круговой буфер)
        if (matrixIndex >= matrixPool.length) {
            matrixIndex = 0; 
        }
        Matrix4f mat = matrixPool[matrixIndex++];
        return mat.identity(); // Сбрасываем матрицу в дефолтное состояние
    }

    // Вызывается в самом начале отрисовки каждого кадра
    public void reset() {
        this.matrixIndex = 0; // Просто сдвигаем указатель в начало!
    }
}
```

А теперь используем наш пул в коде рендеринга сущностей:

```java
public void renderEntity(Entity entity, RenderContext ctx) {
    // ВМЕСТО new Matrix4f() берем "самокат" из пула:
    Matrix4f modelMatrix = ctx.getMatrix(); // 0 аллокаций в куче!
    
    modelMatrix.translate(entity.position);
    shader.setMatrix4f("uModel", modelMatrix);
    
    // В конце кадра вызовется ctx.reset(), и все матрицы снова будут готовы к работе!
}
```

Мы не выделяем память, не гоняем GC. Мы просто бесконечно переиспользуем готовые объекты по кругу.

## 6. Плоские массивы и Битовые сдвиги: как обмануть JVM

Давай пойдем еще глубже. Допустим, нам нужно написать алгоритм расчета освещения вокселей (как в Minecraft). Свет распространяется во все стороны от факела на 15 блоков.
Обычно программисты пишут красивый ООП-код (ООП — объектно-ориентированное программирование): создают класс-узел `LightNode` с полями `int x, y, z, level` и запускают поиск в ширину (BFS) через очередь `Queue<LightNode> queue = new LinkedList<>()`.

Если в мире ломается блок, этот алгоритм обходит тысячи соседних вокселей. На каждый шаг создается `LightNode`, а в очереди `LinkedList` создаются внутренние объекты `Node` для связывания списка. **Это миллионы аллокаций в секунду!** Производительность игры падает до нуля.

### Как эту задачу решает Zenith?

Мы полностью отказались от объектов `LightNode`! Вместо этого мы используем **один плоский примитивный массив `long[]`** в качестве очереди и упаковываем все 3D-координаты и уровень света в одно 64-битное число `long`!

Давай посмотрим, как работает эта битовая магия:

```
        ==========================================================
        |                 Число LONG (64 бита)                   |
        +---------------+---------------+----------------+-------+
        |    Коорд. X   |   Коорд. Y    |    Коорд. Z    | Свет  |
        |   (26 бит)    |    (8 бит)    |    (26 бит)    |(4 бита)
        ==========================================================
```

```java
// Упаковываем X, Y, Z и уровень света в один long (64 бита)
public static long packLightNode(int x, int y, int z, int lightVal) {
    return ((long) (x & 0x3FFFFFF) << 38) |  // Сдвигаем X влево на 38 бит
           ((long) (y & 0xFF) << 30) |        // Сдвигаем Y влево на 30 бит
           ((long) (z & 0x3FFFFFF) << 4) |   // Сдвигаем Z влево на 4 бита
           (lightVal & 0xF);                  // Записываем свет в последние 4 бита
}

// А вот так легко достаем обратно (распаковка):
public static int unpackX(long node) {
    return (int) (node >> 38); // Просто двигаем биты обратно вправо!
}

public static int unpackLight(long node) {
    return (int) (node & 0xF); // Маской зануляем всё, кроме последних 4 бит
}
```

* **Почему это работает так быстро?**
  1. `long[]` — это плоский одномерный массив чисел прямо в памяти. Для Java здесь вообще **нет объектов**. Сборщик мусора просто проходит мимо, так как чистить нечего.
  2. Битовые сдвиги (`<<`, `>>`) и побитовое ИЛИ (`|`) выполняются процессором за **1 такт** (суб-наносекунда). Это одни из самых быстрых операций в вычислительной технике.

## 7. Кэш процессора (L1/L2/L3) и Локальность Данных (Data Locality)

Теперь заглянем под капот самого железа. Чтобы писать быстрый код, нужно понимать физику процессора.

Представь, что ты сидишь за письменным столом и пишешь курсовую работу:
1. **Регистры процессора (Registers):** Это ручка в твоей руке. Ты пишешь мгновенно (1 такт).
2. **L1-кэш (L1 Cache):** Это твой открытый блокнот на столе. Чтобы прочесть слово, тебе нужно просто опустить взгляд (4 такта).
3. **L3-кэш (L3 Cache):** Это книга на полке шкафа в твоей комнате. Нужно встать и подойти к шкафу (40 тактов).
4. **RAM (Оперативная память):** Это книга, которая лежит **в библиотеке на другом конце города**! Чтобы прочесть из неё одно слово, тебе нужно одеться, сесть на автобус и ехать через пробки (**200-300 тактов процессора!**).

Если процессору приходится постоянно бегать в оперативную память (RAM) за данными, он простаивает без дела 99% своего времени. Это называется **Cache Miss (кэш-промах)**.

### Проблема указателей в Java (Pointer Chasing)
В Java все объекты — это указатели. Если ты создаешь массив объектов блоков `Block[]`, в памяти это выглядит как цепочка стрелочек:
`Массив ссылок -> [Адрес 1, Адрес 2, Адрес 3] --> Объект Block лежит на другом конце RAM!`

Когда процессор пытается прочитать этот массив, ему приходится прыгать по ссылкам в случайные места оперативной памяти. Кэш процессора становится бесполезным, постоянно происходят кэш-промахи, и производительность резко падает.

### Решение Zenith: Локальность Данных
Процессор устроен умно. Когда ты просишь его прочитать одну ячейку из RAM, он привозит из библиотеки не одно число, а сразу целую коробку из **64 последовательных байт** (это называется **Cache Line**). А еще он пытается предугадать твои шаги: если ты читаешь массив чисел последовательно, он заранее подгружает следующие блоки в L1-кэш (**Hardware Prefetching**).

Именно поэтому в Zenith все данные воксельного чанка размером 16x16x16 хранятся в виде **одного плоского одномерного массива примитивов `int[]`**:

```java
// Все 4096 блоков чанка лежат в памяти строго друг за другом
private final int[] blockData = new int[16 * 16 * 16];

public int getBlockType(int x, int y, int z) {
    // Быстро переводим 3D координаты в одномерный индекс сдвигами
    int index = (y << 8) | (z << 4) | x;
    return blockData[index]; // Мгновенное чтение!
}
```

Поскольку весь чанк лежит в памяти в виде одного непрерывного физического куска, процессор считывает первый блок и автоматически загружает в L1-кэш данные о следующих 16 блоках. Кэш-промахи падают до нуля, скорость работы возрастает в десятки раз.

Давай закрепим материал в виде каверзных вопросов, которые любят задавать на собеседованиях на позиции Middle/Senior Java & Game Developer, и правильных «матерых» ответов на них.

### Вопрос 1: Что такое GC Pause (Stop-the-World) и как свести её к минимуму в высоконагруженных UI/Game циклах (UI — пользовательский интерфейс)?
* **Ответ «для чайников»:** GC Pause возникает, когда сборщик мусора останавливает всю программу, чтобы спокойно прибраться в куче (Heap) и выбросить ненужные объекты. Если уборка затягивается дольше 8-16 мс, игра пропускает кадры, и игрок видит лаги.
* **Как бороться:** В критических горячих циклах (Hot Loops) нужно соблюдать стратегию **Zero-Allocation**:
  1. Запретить оператор `new` в игровом цикле.
  2. Использовать **Object Pooling (Пулы объектов)** для временных структур (вектора, матрицы).
  3. Хранить данные в плоских массивах примитивов (`int[]`, `long[]`), упаковывая структуры битовыми сдвигами, чтобы не создавать объекты-обертки.

### Вопрос 2: Как работает Escape Analysis в JVM и почему на него нельзя слепо полагаться?
* **Ответ «для чайников»:** Escape Analysis — это когда умный JIT-компилятор (JIT — компиляция "на лету" / Just-In-Time) смотрит: *«Ага, созданный вектор не выходит за рамки этого метода. Зачем мне создавать его в куче и мусорить? Давай я просто разложу его на три локальных float прямо в стеке!»* (это называется *Scalar Replacement*).
* **Почему нельзя полагаться:** 
  1. Оптимизация включается только в рантайме после прогрева кода. Первые секунды мусор всё равно будет лететь в кучу.
  2. Если твой метод слишком длинный, JIT-компилятор откажется его встраивать (Inline), и Escape Analysis мгновенно сломается.
  3. Стоит тебе передать этот объект в любой сторонний метод, записать в массив или поле класса — объект «убегает», оптимизация выключается, и он улетает мусорить в кучу.

### Вопрос 3: В чем разница в производительности между итерацией по `int[]` и `Integer[]`?
* **Ответ «для чайников»:** В плоском `int[]` все числа лежат в памяти ровно и последовательно. Процессор считывает их пачками (Cache Line) в сверхбыстрый L1-кэш без промедлений. В `Integer[]` массив хранит лишь ссылки (указатели) на объекты `Integer`, раскиданные по случайным местам кучи. Процессору приходится постоянно прыгать по адресам RAM (Pointer Chasing), что вызывает жуткие кэш-промахи и заставляет его простаивать сотни тактов в ожидании данных.
