# Глава 6. Рендеринг интерфейсов & Шейдерная анимация шрифтов

После тяжёлой физики — самое время заняться визуальной эстетикой движка.

В этой главе разберём, как устроен рендеринг пользовательского интерфейса (UI) и шрифтов в **Zenith**.

Конкретно: как верстать интерфейсы декларативно с помощью JSON-конфигов, не хардкодя координаты кнопок в коде, как работает аппаратное отсечение пикселей в списках с помощью «картонного трафарета» **glScissor** и как с помощью простейшей тригонометрии превратить обычный текст в живую шейдерную анимацию (радугу, волну и тряску).

## 1. Декларативная верстка GUI (Graphical User Interface — графический интерфейс пользователя): забываем про пиксели

В старых графических движках координаты каждого элемента (кнопок, слотов инвентаря, картинок) прописывали прямо в коде:
`drawSlot(x + 120, y + 45);`

Проблема очевидна: стоило игроку изменить разрешение экрана или масштаб интерфейса, как все элементы расползались по монитору, накладывались друг на друга, а кнопки улетали за пределы экрана.

В Zenith вся верстка GUI вынесена в **декларативные JSON-файлы** (папка `gui/`). Код Java занимается только логикой-контроллером, а всю математику позиционирования берет на себя движок.

```
       JSON макет                 Расчет разметки              Результат на экране
  +------------------+         +--------------------+         +-----------------+
  | "slots": 9,      |  ---->  | Relative Anchors   |  ---->  |  [■][■][■][■]   |
  | "anchor": "center"|        | "slots" units calc |         |  [■][■][■][■]   |
  +------------------+         +--------------------+         +-----------------+
```

### Ключевые фичи разметки Zenith:
* **Слотовые единицы измерения (символ "s"):**
  Координаты элементов задаются не в пикселях, а в размерах ячеек инвентаря. Запись `"x": "3s"` означает: *«смести этот элемент на ширину 3 стандартных слотов инвентаря с учетом всех зазоров между ними»*.
* **Относительные якоря (`relativeTo`):**
  Элементы жестко привязываются друг к другу. Допустим, мы привязали слоты брони к левой грани окна инвентаря. Если игрок сдвинет окно инвентаря, слоты брони послушно сдвинутся вслед за ним, сохранив идеальные отступы!
* **Адаптивные фоны (`includeGroups`):**
  Стеклянные размытые панели на заднем плане не имеют фиксированного размера. Движок сканирует координаты всех активных элементов, вычисляет их общий охватывающий прямоугольник (Bounding Box) и автоматически натягивает стеклянный фон ровно под этот размер.

## 2. Аппаратное отсечение пикселей через `glScissor`

Когда ты прокручиваешь список статей в «Дневнике Выжившего» (Survivor's Tablet), текст и картинки при прокрутке вверх или вниз должны красиво исчезать на границах книги, а не вылезать за её поля.

Для этого используется аппаратный тест отсечения OpenGL — **Scissor Test (Ножничный тест)**.

```java
GL11.glEnable(GL11.GL_SCISSOR_TEST); // 1. Включаем ножничный тест
GL11.glScissor(x, y, width, height);  // 2. Вырезаем прямоугольную маску
// ... Рисуем наш прокручиваемый список статей ...
GL11.glDisable(GL11.GL_SCISSOR_TEST); // 3. Выключаем тест
```
Любой пиксель, который рендерер попытается нарисовать за пределами прямоугольника `glScissor`, будет аппаратно проигнорирован видеокартой на стадии растеризации.

### ⚠️ Критическая ловушка Y-координаты!
Вот тут спотыкаются 99% новичков.

* В нашей 2D UI-системе точка начала координат $(0,0)$ находится в **левом верхнем углу** экрана, а ось Y направлена **вниз**.
* В `glScissor` в OpenGL точка начала координат $(0,0)$ находится в **левом нижнем углу** физического окна, а ось Y направлена **вверх**.

```
    UI Координаты (Y-Down)                   OpenGL Scissor (Y-Up)
    (0,0) +------------------+                 +------------------+ (width, height)
     │                       |                 |  glScissor Box   |
     │   +--------------+    |                 |   +----------+   |
     │   | Панель UI    |    |                 |   |          |   |
     ▼   +--------------+    |                 |   +----------+   |
                             |                 (0,0) +------------+
```

Если ты передашь координаты панели UI напрямую в `glScissor`, зона отсечения окажется перевернутой «вверх ногами» и обрежет совсем не то, что нужно!

В `ScrollPanel.java` мы выполняем математическое преобразование координат с учетом высоты окна игрока `windowHeight` и масштаба DPI (Dots Per Inch — плотность пикселей на дюйм) (`scaleFactor`):

```java
// 1. Масштабируем координаты под DPI монитора (для 4K / Retina экранов)
int scissorX = Math.round(uiPanelX * scaleFactor);
int scissorWidth = Math.round(uiPanelWidth * scaleFactor);
int scissorHeight = Math.round(uiPanelHeight * scaleFactor);

// 2. Инвертируем Y: берем общую высоту окна, вычитаем Y-координату панели и её высоту
int scissorY = Math.round((windowHeight - (uiPanelY + uiPanelHeight)) * scaleFactor);

// 3. Передаем правильные координаты в OpenGL!
GL11.glScissor(scissorX, scissorY, scissorWidth, scissorHeight);
```
Теперь маска отсечения лежит ровно поверх панели, и текст прокручивается корректно.

## 3. Как нарисовать текст в OpenGL

Вывод текста на экран в низкоуровневой графике — нетривиальная задача. Мы не можем просто отправить строку в GPU (Graphics Processing Unit — графический процессор).
1. **Character Atlas (Атлас букв):** Движок заранее генерирует одну большую текстуру, содержащую все буквы алфавита (латиницу, кириллицу, цифры и значки).
2. **Генерация меша:** Для каждой буквы в строке `FontRenderer` создает плоский прямоугольник (квад) из 4 вершин. Текстурные координаты UV (двумерные координаты наложения текстуры) настраиваются так, чтобы они указывали точно на область этой буквы в атласе.
3. Все квадры букв упаковываются в один буфер и рисуются за один единственный Draw Call (команда на отрисовку от процессора к видеокарте).

### Управляющие коды (Minecraft-style)
`FontRenderer` парсит строки на наличие символа `$`.
* `$0` - `$f` — переключают цвета букв.
* `$l` — включает жирный шрифт (делает микро-смещение координат букв по X, рисуя их дважды).
* `$o` — включает курсив (применяет шейдерный скос матрицы модели вбок).
* `$r` — сбрасывает все эффекты.

## 4. Живые шейдерные эффекты текста в `ui_fragment.glsl`

Zenith поддерживает анимированные текстовые эффекты. Во фрагментный шейдер `ui_fragment.glsl` передаются специальные битовые флаги анимаций. Разберём их математику.

### 4.1 Эффект Радуги (`$z` — Rainbow)
Цвет букв плавно переливается всеми цветами радуги. Мы вычисляем сдвиг цветового тона (Hue) на основе координаты X буквы и системного времени `uTime`:
```glsl
if ((uTextEffects & EFFECT_RAINBOW) != 0u) {
    // fract() оставляет только дробную часть числа, удерживая тон в диапазоне 0.0 - 1.0
    float hue = fract(uTime * 0.2 + vLocalPos.x * 0.005);
    
    // Переводим тон в RGB (Red, Green, Blue — цветовая модель) цвет
    vec3 rgb = hsv2rgb(vec3(hue, 0.8, 1.0));
    FragColor = vec4(rgb, 1.0) * texture(uTexture, vTexCoords);
}
```

### 4.2 Эффект Волны (`$v` — Wave)
Буквы плавно плывут вверх-вниз по синусоидальной волне. Этот расчет происходит в **вершинном шейдере**:
```glsl
if ((uTextEffects & EFFECT_WAVE) != 0u) {
    // Используем синус от времени и координаты X буквы!
    float waveOffset = sin(uTime * 4.0 + aLocalPos.x * 0.05) * 4.0; // Амплитуда 4 пикселя
    finalPosition.y += waveOffset;
}
```

### 4.3 Эффект Тряски (`$q` — Shake)
Буквы мелко и хаотично вибрируют, создавая эффект страха или паники (используется при критическом износе инструментов):
```glsl
if ((uTextEffects & EFFECT_SHAKE) != 0u) {
    // Генерируем быструю тряску на основе синуса и косинуса от времени
    float shakeX = sin(uTime * 50.0 + aLocalPos.y) * 0.8;
    float shakeY = cos(uTime * 45.0 + aLocalPos.x) * 0.8;
    finalPosition.xy += vec2(shakeX, shakeY);
}
```

## 5. Matrix Crosshair: прицел, который умеет дышать

### Суть проблемы на пальцах
Обычный прицел в играх — статическая картинка (текстура). В кубическом мире Zenith это выглядит бедно. Прицел должен реагировать на действия игрока: плавно расширяться при беге, сжиматься в точку при прицеливании и пульсировать при наведении инструмента на слабую точку блока (Weak Spot).

Передавать сотни текстур под каждое мизерное изменение — безумие для видеопамяти. Поэтому в Zenith используется **Matrix Crosshair** — прицел, который рисуется прямо на видеокарте с помощью математики знаковых полей расстояний (**SDF — Signed Distance Fields**).

### Метафора из жизни
Вспомни зрачок живого существа. В темноте или при испуге (бег) он расширяется, чтобы уловить больше света и информации. При ярком свете или концентрации на микроскопической детали (наведение на цель) он сжимается в острую точку. Наш прицел — это такой же математический зрачок, реагирующий на физику персонажа.

### GLSL Шейдер прицела
Рассмотрим фрагментный шейдер прицела, рисующий гладкое кольцо без единой текстуры:

```glsl
#version 330 core
out vec4 FragColor;
in vec2 TexCoords; // Локальные координаты прицела от -1.0 до 1.0

uniform float uRadius;     // Динамический радиус кольца (меняется от скорости игрока)
uniform float uThickness;  // Толщина линии прицела
uniform vec3 uColor;       // Цвет прицела (например, зеленый или красный при перегреве)

void main() {
    // Шаг 1: Находим расстояние от текущего пикселя до центра прицела (0.0, 0.0)
    float dist = length(TexCoords);
    
    // Шаг 2: Магическая SDF-формула кольца.
    // Вычитаем из расстояния радиус и получаем отклонение. 
    // abs() делает отклонение всегда положительным, а вычитание толщины uThickness
    // формирует саму "трубку" кольца.
    float sdf = abs(dist - uRadius) - uThickness;
    
    // Шаг 3: Сглаживание краев (Antialiasing) «на лету».
    // Функция smoothstep плавно переводит альфа-канал от 1.0 до 0.0 на границе пикселей,
    // полностью убирая «лесенки» (пикселизацию).
    float alpha = smoothstep(0.015, 0.0, sdf);
    
    // Шаг 4: Выводим итоговый пиксель с учетом прозрачности
    FragColor = vec4(uColor, alpha);
}
```

**Разбор кода «на пальцах»:**
1. `length(TexCoords)` считает расстояние по теореме Пифагора: $d = \sqrt{x^2 + y^2}$. Центр экрана имеет координаты $(0,0)$.
2. Формула `abs(dist - uRadius) - uThickness` возвращает отрицательное значение для пикселей, находящихся *внутри* тела кольца, и положительное — *снаружи*.
3. `smoothstep` берет эту разницу и превращает её в плавный градиент прозрачности. Результат — ровные, мягкие края при любом разрешении экрана.

## 6. Minimap Radar: тактический сонар на экране

### Суть проблемы на пальцах
В кубическом мире Zenith звук играет ключевую роль. Монстры слышат ваши шаги, работу ваших механизмов и рубку деревьев. Чтобы игрок визуально понимал, насколько он шумит и откуда исходит потенциальная угроза, на экран выводится тактический радар-сонар. 

Он работает полностью на процедурных шейдерных эффектах: испускает расходящиеся круги звуковых волн, которые при соприкосновении с источниками шума подсвечивают их яркими импульсными метками.

### Метафора из жизни
Представь тихий пруд ночью. Ты бросаешь в воду камень — от него расходятся идеальные концентрические круги. Если на поверхности воды плавает поплавок (источник шума/монстр), проходящая волна заколышет его, заставив выдать ответные мелкие волны. Радар делает то же самое на экране вашего планшета.

### GLSL Шейдер процедурного сонара
Посмотрим, как видеокарта рисует волну сонара и отметку источника шума:

```glsl
// fragment_radar.glsl
#version 330 core
out vec4 FragColor;
in vec2 uV; // Координаты текстуры радара от -1.0 до 1.0 (центр в 0.0)

uniform float uTime;          // Глобальное время для анимации движения волн
uniform vec2 uNoiseSource;    // Положение источника звука (например, крадущегося монстра)
uniform float uPlayerNoise;   // Текущий уровень шума игрока (радиус расхождения волны)

void main() {
    // Шаг 1: Вычисляем расстояние от центра радара
    float distToCenter = length(uV);
    
    // Шаг 2: Генерируем круговую волну, бегущую от центра
    // mod(uTime, 1.0) зацикливает анимацию волны от 0 до 1
    float waveRadius = mod(uTime * 0.8, 1.0) * uPlayerNoise;
    
    // Формируем тонкую светящуюся полосу волны сонара
    float wave = smoothstep(0.03, 0.0, abs(distToCenter - waveRadius));
    
    // Шаг 3: Вычисляем расстояние от текущего пикселя до источника шума (монстра)
    float distToMonster = length(uV - uNoiseSource);
    
    // Создаем яркую круглую точку для монстра
    float monsterMarker = smoothstep(0.06, 0.0, distToMonster);
    
    // Шаг 4: Метка монстра становится видимой ТОЛЬКО тогда, когда её настигает волна!
    float isSwept = step(distToCenter, waveRadius + 0.05); 
    float finalMonsterIntensity = monsterMarker * isSwept * (1.0 - waveRadius);
    
    // Смешиваем цвета: неоновый зеленый для волны радара + тревожный красный для монстра
    vec3 waveColor = vec3(0.0, 1.0, 0.4) * wave * 0.4;
    vec3 monsterColor = vec3(1.0, 0.1, 0.1) * finalMonsterIntensity * 1.0;
    
    float finalAlpha = wave * 0.4 + finalMonsterIntensity * 0.9;
    
    FragColor = vec4(waveColor + monsterColor, finalAlpha);
}
```

**Разбор кода «на пальцах»:**
1. Выражение `abs(distToCenter - waveRadius)` вычисляет расстояние до гребня бегущей волны. С помощью `smoothstep` мы заставляем светиться только узкую полоску на этом гребне.
2. Функция `step(a, b)` работает как жесткий переключатель: «волна сонара дошла до цели — показать красную точку».
3. Множитель `(1.0 - waveRadius)` постепенно гасит интенсивность засвета по мере удаления волны от эпицентра.

## 7. Survivor's Tablet: анатомия выживальческой документации

Планшет выжившего (**Survivor's Tablet**) — сложный модульный интерфейс, построенный на чистом OpenGL. Разберём его ключевые узлы.

### 7.1 ScrollPanel и магия glScissor
Панель прокрутки в Zenith использует функцию `glScissor` для аккуратной обрезки содержимого.

**Аналогия:** Представьте картонную маску с прорезанным в ней квадратным окошком. Под этой маской можно двигать длинный свиток с текстом — в окошке будет видна только текущая строчка, а всё остальное скроется за картоном. `glScissor` делает то же самое — указывает видеокарте прямоугольную область на экране, за пределами которой пиксели просто запрещено рисовать.

**Нюанс:** В оконной системе рабочего стола начало координат $(0,0)$ находится в **левом верхнем** углу, а ось Y растёт вниз. В OpenGL же начало координат $(0,0)$ расположено в **левом нижнем** углу экрана, и Y растёт вверх. Поэтому при вызове `glScissor` приходится программно переворачивать Y.

```java
/**
 * Отрисовывает контент панели, жестко обрезая его по границам ScrollPanel.
 */
public void renderScissoredContent(int x, int y, int width, int height, float scrollY) {
    // 1. Включаем аппаратный тест отсечения пикселей в OpenGL
    GL11.glEnable(GL11.GL_SCISSOR_TEST);

    // 2. Получаем текущую высоту игрового окна
    int windowHeight = Display.getHeight();

    // 3. Перевод Y из оконной системы координат в систему OpenGL.
    // Вычитаем y + height из полной высоты экрана, чтобы найти нижнюю границу окна отсечения!
    int scissorY = windowHeight - (y + height);

    // 4. Задаем прямоугольную маску отсечения
    GL11.glScissor(x, scissorY, width, height);

    // 5. Сдвигаем матрицу рендеринга на величину скролла и рисуем длинную простыню текста
    GL11.glPushMatrix();
    GL11.glTranslatef(0.0f, -scrollY, 0.0f);
    
    this.drawTextAndElements();

    GL11.glPopMatrix();

    // 6. Обязательно выключаем тест отсечения, иначе весь остальной интерфейс игры тоже обрежется!
    GL11.glDisable(GL11.GL_SCISSOR_TEST);
}
```

### 7.2 Dynamic Recipes (Интерактивные рецепты в планшете)
Интерактивные рецепты генерируются на лету:

1. **Stump Rendering**: Движок считывает данные рецепта, находит инструмент и исходный блок, после чего отображает их с необходимыми текстовыми индикаторами.
2. **Napping Rendering**: Отрисовывает виртуальную сетку $5\times5$ пикселей, визуализируя процесс обработки материала. Если рецепт поддерживает вариативность материалов, иконки в схеме циклически меняются, демонстрируя доступные альтернативы.

## 8. Готовимся к IT-собеседованию

### Вопрос 1: Как работает scissor-тест (`glScissor`) в OpenGL? Чем он отличается от `glViewport`?
* **Ответ «для чайников»:** `glScissor` работает как трафаретная картонная маска: он задает прямоугольную область, и всё, что рисуется вне её, видеокарта аппаратно отбрасывает (обрезает края картинки без изменения масштаба). `glViewport` же задает область сжатия/масштабирования координат: если уменьшить viewport, вся сцена сожмется и уместится внутри указанной рамки.

### Вопрос 2: Как синхронизировать координаты при использовать `glScissor` в графических интерфейсах и зачем нужен `scaleFactor`?
* **Ответ «для чайников»:** Сложность в том, что в UI начало координат находится в левом верхнем углу (ось Y вниз), а в `glScissor` — в левом нижнем углу (ось Y вверх). Для синхронизации мы инвертируем Y: `scissorY = WindowHeight - (UI_Y + UI_Height)`. 
* Параметр `scaleFactor` обязателен, чтобы учесть плотность пикселей на дисплеях с высоким DPI (Retina / 4K), где физический размер буфера кадра в пикселях больше виртуальных координат UI.

### Вопрос 3: Как реализовать шейдерный перелив радуги для текста?
* **Ответ «для чайников»:** При генерации меша строки мы записываем локальное смещение каждой буквы от начала строки в атрибут вершины `aLocalPos.x`. В фрагментном шейдере рассчитываем цветовой тон Hue в диапазоне 0.0 - 1.0, плавно меняя его во времени: `hue = fract(uTime * Speed + vLocalPos.x * Frequency)`. Полученный тон переводим из цветового пространства HSV (Hue, Saturation, Value — цветовая модель) в RGB и умножаем на текстуру буквы.

### Вопрос 4: Какую практическую проблему решает инвертирование координаты Y по формуле Display.getHeight() - (y + height) при настройке glScissor для ScrollPanel в Zenith?
* **Ответ «для чайников»:** Это компенсирует разницу в начале координат: в OpenGL точка (0,0) находится в левом нижнем углу экрана, а в событиях ввода ОС и оконной системе — в левом верхнем. Чтобы аппаратное отсечение произошло ровно в границах нашей ScrollPanel, мы обязаны переводить экранные координаты GUI в пространство OpenGL.
