## 1. Процедурная физика рук: Пружина в банке с мёдом

В классических играх, когда персонаж идёт или резко поворачивает голову, его руки и оружие двигаются по заранее нарисованным анимациям. Но в реальной жизни руки обладают массой — они отстают при рывках, колеблются и плавно догоняют тело.

Чтобы не рисовать миллион текстур под каждую траекторию, в Zenith используется **динамический симулятор физики пружины 2-го порядка (пружина-масса-демпфер)**.

```
       ================================
       |   Камера (Цель пружины)      |
       ================================
                      │
                      │  (Виртуальная пружина: жесткость stiffness)
                      ▼
                  (  / \  )   <-- Банка с мёдом (damping) гасит колебания
                  ( \ / / )
                      ▼
       ================================
       |   Рука с топором (масса mass)|
       ================================
```

### Дифференциальное уравнение пружины «на пальцах»
Физику этого процесса описывает грозное дифференциальное уравнение второго порядка:
$$m \frac{d^2x}{dt^2} + c \frac{dx}{dt} + k(x - x_{target}) = 0$$

Переведём его на человеческий язык:
* $x$ — где рука находится сейчас.
* $x_{target}$ — куда рука должна прийти вслед за движением камеры.
* $m$ — **Mass (Масса/Тяжесть предмета).** 
  Чем тяжелее предмет, тем дольше он разгоняется и тем дольше летит по инерции при резком торможении. Деревянная заточка легкая, а тяжелая кувалда отстает очень сильно.
* $k$ — **Stiffness (Жесткость пружины).**
  С какой силой виртуальная резинка притягивает руку обратно к центру экрана.
* $c$ — **Damping (Демпфирование/Сопротивление).**
  А вот это — наша **виртуальная банка с мёдом**, в которую засунуты руки. Без неё пружина качалась бы вечно от любого шага, укачивая игрока до морской болезни. Мёд гасит колебания, плавно останавливая руку.

### Численное интегрирование в Java (Semi-implicit Euler)
Чтобы компьютер мог рассчитывать это движение 200 раз в секунду, мы переводим уравнение физики в простейший код (Semi-implicit Euler integration):

```java
// com.za.zenith.engine.graphics.model.ViewmodelPhysics.java
public class ViewmodelPhysics {
    public float currentX = 0f;  // Где рука сейчас
    public float velocityX = 0f;  // Скорость руки
    
    public float mass = 1.0f;      // Тяжесть топора/рук
    public float damping = 8.0f;    // Густота мёда (сопротивление)
    public float stiffness = 50.0f; // Жесткость резинки (пружины)

    public void update(float targetX, float deltaTime) {
        // 1. Ограничиваем deltaTime (dt), чтобы при падении FPS (Frames Per Second — количества кадров в секунду) до 5 кадров 
        // наша пружина не улетела в космос из-за гигантского скачка времени.
        float dt = Math.min(deltaTime, 0.05f);
        
        // 2. Сила пружины (Закон Гука): тянем руку к цели
        float springForce = (targetX - currentX) * stiffness;
        
        // 3. Сила трения мёда: тормозим руку пропорционально скорости
        float dampingForce = -velocityX * damping;
        
        // 4. Ускорение (Второй закон Ньютона: F = m*a -> a = F / m)
        float accelerationX = (springForce + dampingForce) / mass;
        
        // 5. Прибавляем ускорение к скорости, а скорость — к координате!
        velocityX += accelerationX * dt;
        currentX += velocityX * dt;
        
        // 6. Защита от деления на ноль и сбоев математики
        if (!Float.isFinite(currentX)) {
            currentX = targetX;
            velocityX = 0f;
        }
    }
}
```
**Результат:** когда игрок берёт тяжёлый топор, движок считывает из JSON (JavaScript Object Notation — текстового формата обмена данными) вес `"weight": 3.0` и увеличивает `mass` в пружине. При повороте камеры топор заметно отстаёт, а затем плавно догоняет прицел, создавая ощущение реального веса оружия в руках.

## 2. Кватернионы: Умные сферические шарниры

В 3D-графике классический способ повернуть модельку — использовать три угла Эйлера: **Pitch (наклон вверх-вниз)**, **Yaw (поворот влево-вправо)** и **Roll (наклон на бок)**.

### Ужас Шарнирного Замка (Gimbal Lock)
Углы Эйлера поворачивают объект последовательно: сначала вокруг оси X, потом Y, потом Z. 

И тут нас подстерегает геометрическая ловушка. Представьте, что рука персонажа наклонена по Pitch строго на 90 градусов вверх. Локальная ось Roll (наклон руки) совпадает с глобальной осью Yaw (поворот влево-вправо).

Две оси вращения накладываются друг на друга. Объект теряет одну из трёх степеней свободы. Вращение в одной плоскости становится невозможным. Стоит игроку попытаться довернуть руку, как модельку начинает швырять и перекручивать во все стороны. Это и есть **Gimbal Lock (Шарнирный замок)**.

```
       ==========================================================
       |                  GIMBAL LOCK                           |
       |                                                        |
       |        Наклонили руку на 90 градусов вверх.            |
       |        Две оси вращения совпали в пространстве.        |
       |        Третья ось заблокирована. Рука дергается!       |
       ==========================================================
```

### Спасение: Кватернионы (`Quaternionf`)
Кватернион — это математический объект из четырех чисел $(x, y, z, w)$. 
Не стоит пугаться формул с мнимыми единицами. По сути, кватернион — это **«умный сферический шарнир»**. Он задаёт вращение сразу вокруг одной произвольной оси на определённый угол в четырёхмерном пространстве.

* **Почему они идеальны для Zenith?**
  1. **Оси никогда не совпадут:** У кватернионов нет последовательного вращения, поэтому Gimbal Lock физически невозможен.
  2. **Идеально плавные переходы (Slerp):** Если нам нужно плавно переключить анимацию рук со стойки бега на удар мечом, кватернионы позволяют делать **Slerp (сферическую линейную интерполяцию)**.

```java
// Сферическое сглаживание углов в Zenith
public static Quaternionf slerp(Quaternionf start, Quaternionf end, float alpha, Quaternionf dest) {
    return start.slerp(end, alpha, dest); // Кратчайший и самый плавный путь по сфере поворотов
}
```
В отличие от углов Эйлера, которые при интерполяции крутят объект по ломаным траекториям, Slerp поворачивает кость по идеальной, кратчайшей траектории на сфере вращений.

## 3. Математика PCA (Метод Главных Компонент) для предметов

Когда персонаж берёт в руки плоский предмет (например, пиксельный топор или кирку), движок автоматически строит его 3D-модель из обычной 2D-картинки. Но как алгоритму понять, под каким углом правильно развернуть топор, чтобы рукоятка легла ровно в ладонь, а не торчала боком сквозь пальцы?

В генераторе мешей `ItemMeshGenerator` мы применили **метод главных компонент (PCA)** — классический алгоритм линейной алгебры и машинного обучения!

```
      Текстура предмета          Облако пикселей            Главные Оси (PCA)
      +--------------+          +--------------+          +--------------+
      |   /          |          |   *          |          |   / e1       |
      |  / Топор     |   --->   |  * *  *      |   --->   |  / (Рукоять) |
      | /            |          | *   *        |          | / ----> e2   |
      +--------------+          +--------------+          +--------------+
```

### Алгоритм PCA на пальцах:
1. **Собираем координаты:** Движок сканирует картинку 16x16. Для каждого непрозрачного пикселя мы получаем координаты точки $(x, y)$.
2. **Центрируем облако:** Находим центр масс предмета (средние координаты) и сдвигаем все точки так, чтобы центр масс оказался в точке `(0, 0)`.
3. **Строим матрицу ковариации 2x2:** Эта матрица показывает, насколько координаты пикселей вытянуты по осям X и Y относительно друг друга (то есть форму предмета).
4. **Ищем Собственные Векторы:** 
   Решаем характеристическое уравнение матрицы и получаем два перпендикулярных вектора $\vec{e}_1$ и $\vec{e}_2$.
   * Вектор $\vec{e}_1$, соответствующий наибольшему числу, указывает направление наибольшей вытянутости облака пикселей. **Это и есть физическое направление рукоятки топора!**
5. **Выравниваем предмет:** Движок рассчитывает угол наклона этого вектора $\vec{e}_1$ и автоматически поворачивает весь 3D-меш предмета вертикально.
6. **Smart Grasping:** Мы находим крайние пиксели вдоль этой оси и привязываем их к суставу кисти игрока. Топор идеально ложится в руку без ручной настройки.

> [!NOTE]
> **PCA без университетского матана**
> Если слова «ковариационная матрица» вызывают у вас панику, то вот перевод на человеческий язык. Ковариационная матрица — это просто способ узнать, насколько кучно пиксели ложатся по диагонали картинки. А «собственный вектор» — это математический термин для линии наибольшей симметрии. Для нарисованного топора эта линия всегда пройдет точно вдоль его рукояти. Алгоритм просто находит самую длинную «ось» облака пикселей и ориентирует предмет по ней!

## 4. Скелетный Inverse Kinematics (FABRIK IK): Метод сборщика бус

Как сделать так, чтобы ноги монстров реалистично стояли на неровных склонах гор, а пальцы персонажа цеплялись за края блоков при паркуре? 

Для этого используется **Инверсная Кинематика (IK)** — мы указываем точку цели (куда должна прийти рука/нога), а компьютер сам рассчитывает углы сгиба суставов.

В Zenith встроен передовой алгоритм **FABRIK** (Forward And Backward Reaching Inverse Kinematics).

```
   1. Исходное        2. Backward Pass (К цели)       3. Forward Pass (К базе)
      База               База                             База (Привязана!)
      O                   O                                O
       \                   \                                \
        \                   \                                \
         O                   O                                O
          \                   \                                \
           \ Target            \ Target (Сошлась!)             \ Target
            X                   O                               O
```

Большинство старых алгоритмов IK используют тяжёлую тригонометрию и обратные матрицы Якобиана. Они медленные и часто «взрывают» суставы, выворачивая локти наизнанку.

**FABRIK работает исключительно на простейшей геометрии векторов.** Это метод сборщика бус.

Представьте веревку с бусинами (костями плеча, локтя и кисти). Нам нужно дотянуть конец веревки до мухи на стене (цели).

### Шаг 1: Backward Pass (Обратный ход)
Мы берем последнюю бусину (кисть) и насильно ставим её в точку цели.
Предыдущая бусина (локоть) кричит: *«Эй, расстояние между нами нарушено! Кость же не может растянуться!»*. Мы двигаем локоть по направлению к кисти на фиксированное расстояние длины предплечья.
Повторяем это до самого плеча. Теперь веревка натянулась к мухе, но плечо оторвалось от тела.

### Шаг 2: Forward Pass (Прямой ход)
Мы берем плечо и насильно возвращаем его обратно на тело (в базу сустава).
Теперь локоть кричит: *«Эй, плечо вернулось, расстояние опять сломано!»*. Мы сдвигаем локоть по направлению к плечу на фиксированное расстояние кости плеча.
Повторяем до кисти.

Мы прогоняем эти два шага вперёд-назад всего **3–4 раза**. За микросекунду цепочка суставов сходится в идеальную, естественную позу с точностью до доли миллиметра. Локти сгибаются плавно и красиво.

## 4.5 Entity Physics & Interpolation: Борьба с микро-дерганьем (prevPosition)

### В чём проблема?
Физический движок должен просчитывать столкновения, гравитацию и перемещения с фиксированной частотой (например, 20 или 170 тиков в секунду, они же Fixed Update), чтобы игра работала одинаково стабильно на любом железе. Но если отрисовывать персонажей и монстров с этой же частотой, получится дёрганое слайд-шоу. Рендерер-то выдаёт 144+ кадров в секунду (Variable FPS).

### Метафора из жизни: Контролер в автобусе
Представьте, что вы едете в автобусе, а строгий контролер проверяет ваш билет раз в 5 минут (это наш физический тик). Если бы автобус двигался только в моменты проверок, он бы телепортировался с остановки на остановку рывками. Но на самом деле между проверками автобус едет плавно. Вот и наш рендерер берет на себя роль «плавного водителя», сглаживая путь между двумя проверками-тиками.

Каждая сущность в **Zenith** помнит не только то, где она находится прямо сейчас (`currentPosition`), но и то, где она была ровно один физический тик назад (`prevPosition`). А рендерер, зная, сколько времени прошло между физическими тиками (коэффициент `alpha`), плавно дорисовывает путь сущности прямо «на лету» (линейная интерполяция, Lerp).

### Код «на пальцах»

```java
/**
 * Вычисляет сглаженную позицию сущности для текущего кадра отрисовки.
 * Избавляет наши глаза от дерганья на частотах 144Hz+.
 */
public Vector3f getInterpolatedPosition(Entity entity, float alpha) {
    // 1. Берем позицию сущности на предыдущем физическом тике
    Vector3f prev = entity.getPrevPosition();
    
    // 2. Берем позицию сущности на текущем физическом тике
    Vector3f current = entity.getPosition();
    
    // 3. Вычисляем дельту движения (направление и расстояние)
    float deltaX = current.x - prev.x;
    float deltaY = current.y - prev.y;
    float deltaZ = current.z - prev.z;
    
    // 4. Умножаем дельту на alpha (коэффициент прогресса между тиками от 0.0 до 1.0)
    // и прибавляем к предыдущей позиции. Это и есть линейная интерполяция (LERP)!
    float interpX = prev.x + deltaX * alpha;
    float interpY = prev.y + deltaY * alpha;
    float interpZ = prev.z + deltaZ * alpha;
    
    // Возвращаем идеальную, плавную точку в пространстве
    return new Vector3f(interpX, interpY, interpZ);
}
```

## 4.6 VoxelShape (AABB & TextureAABBGenerator): Коробки для обуви и сканеры

### В чём проблема?
Как объяснить игре, что упавшее на землю яблоко — это круглый фрукт, а не целый кубический метр кирпича? Для этого используются **VoxelShape** — наборы невидимых физических коробочек (`AABB` — Axis-Aligned Bounding Box), которые описывают точную форму блока. 

### Метафора из жизни: Упаковка фарфоровой вазы
Представьте, что вы упаковываете дорогую фарфоровую вазу сложной формы для отправки почтой. Чтобы она не болталась в коробке, вы обкладываете её маленькими прямоугольными пенопластовыми блоками со всех сторон. Для почтовой службы ваша ваза теперь — это просто набор этих прямоугольников. Вот и физический движок видит мир как скопление таких удобных коробочек!

Более того, в **Zenith** реализован `TextureAABBGenerator`. Система способна самостоятельно просканировать текстуру плоского предмета (например, веточки или кремня), отбросить все прозрачные пиксели и построить точный физический хитбокс.
А алгоритм **Raycast** (луч из глаз игрока) честно проверяет столкновение с каждой отдельной коробочкой внутри формы. Благодаря этому вы можете прострелить стрелой пустоту под ступенькой или схватить предмет прямо через дыру в заборе.

### Код «на пальцах»

```java
/**
 * Автоматически генерирует физические границы AABB по пикселям текстуры,
 * игнорируя пустоту (прозрачные пиксели).
 */
public AABB generateAABBFromTexture(Texture texture) {
    int minX = texture.getWidth();
    int maxX = 0;
    int minY = texture.getHeight();
    int maxY = 0;
    
    // Проходимся по каждому пикселю нашей картинки вдоль и поперек
    for (int y = 0; y < texture.getHeight(); y++) {
        for (int x = 0; x < texture.getWidth(); x++) {
            // Если пиксель не прозрачный (альфа-канал больше нуля)
            if (texture.getPixelAlpha(x, y) > 0) {
                // Сужаем рамки нашего физического бокса под реальные границы рисунка
                if (x < minX) minX = x;
                if (x > maxX) maxX = x;
                if (y < minY) minY = y;
                if (y > maxY) maxY = y;
            }
        }
    }
    
    // Переводим пиксельные координаты (например, от 0 до 16) 
    // в относительные координаты воксельного мира (от 0.0f до 1.0f)
    return new AABB(
        minX / 16.0f, minY / 16.0f, 0.0f, // Минимальная точка (X, Y, Z)
        maxX / 16.0f, maxY / 16.0f, 1.0f  // Максимальная точка (X, Y, Z)
    );
}
```

## 4.7 Double Plants: Костюм-двойка для флоры

### В чём проблема?
Как сделать высокую двухметровую рожь или пышный куст роз, не раздувая базу данных новыми уникальными блоками? 

### Метафора из жизни: Брючный костюм-двойка
Пиджак и брюки идут под одним артикулом в магазине, но брюки мы надеваем вниз, а пиджак — наверх. Было бы странно носить два пиджака или две пары брюк одновременно.

В **Zenith** это решено через тип размещения `PlacementType.DOUBLE_PLANT`. Блок физически занимает одну ячейку, но имеет два сегмента с метаданными: `0` для нижней половины и `1` для верхней. В конфигурационном JSON-файле прописывается поле `upperTexture`, и рендерер на лету подменяет текстуру для верхнего сегмента. Один ID блока — два разных визуальных объекта. Система смотрит на метаданные (0 или 1) и понимает, какую деталь нужно применить к этому конкретному вокселю.

### Код «на пальцах»

```java
/**
 * Выбирает правильную текстуру для растения в зависимости от его сегмента (верх/низ).
 */
public Texture getPlantTexture(BlockDefinition block, int metadata) {
    // Если это действительно двухсегментное растение
    if (block.getPlacementType() == PlacementType.DOUBLE_PLANT) {
        // Метаданные 1 говорят о том, что перед нами верхушка куста/травы
        if (metadata == 1) {
            // Возвращаем специальную текстуру для верхней части
            return block.getUpperTexture(); 
        }
    }
    // В любых других случаях (или если это корень, метаданные 0) отдаем базовую текстуру
    return block.getBaseTexture();
}
```

## 5. Физика паркура и синхронизация IK (Ledge Grab Snapping)

Как сделать так, чтобы персонаж не просто упирался лбом в стену, а мог прыгнуть, зацепиться руками за край уступа, повиснуть и подтянуться? И как сделать так, чтобы его ладони при этом честно держались за трёхмерную грань вокселя, а не висели в воздухе?

В Zenith за это отвечает связка физического движка коллизий AABB, контроллера `ParkourHandler` и решателя **FABRIK IK**.

### 5.1 Обнаружение уступа: Двойной зонд
Когда персонаж летит к стене, физический движок запускает алгоритм **Ledge Grab Detection (Поиск уступов)**:
1.  **Simple Voxel Probe:** Прямо перед игроком на уровне груди и глаз выкидываются два коротких физических луча-пробника (Probes).
2.  **Условие зацепа:** Если нижний пробник уперся в плотный воксель, а верхний пробник пролетел сквозь воздух - значит, перед нами идеальный уступ (край блока)!
3.  **Ledge Grab Snapping (Магнит):** Физический движок плавно примагничивает тело игрока к стене:
    *   Позиция персонажа выравнивается строго параллельно плоскости грани блока.
    *   Это убирает баг «прохождения сквозь текстуры», если игрок смотрит на стену под косым углом.

### 5.2 Интеграция стамины и физики
В момент зацепа физика игрока кардинально меняется:
*   Вертикальная скорость обнуляется, гравитация **полностью отключается**! Персонаж переходит в состояние висения (`HANGING`).
*   Система считывает стоимость удержания из JSON-файла действия `actions/hang.json`. Каждую секунду у игрока отнимается выносливость (стамина).
*   Если стамина падает до `0`, гравитация мгновенно включается обратно, и персонаж срывается вниз.

### 5.3 Передача цели в FABRIK IK
Здесь происходит синхронизация визуала и физики.
1.  Физический движок находит точные 3D-координаты края уступа (это две точки в мире, где должны находиться левая и правая ладони персонажа).
2.  Мы берем эти координаты и прогоняем их через матрицу вида (View-Space) в координаты скелета рук.
3.  Эти точки становятся **мировыми целями (Targets)** для нашего решателя **FABRIK IK**!
4.  FABRIK мгновенно вычисляет сгиб плеч и локтей персонажа так, чтобы его 3D-ладони встали ровно на край кубика.
5.  Если игрок двигается боком вдоль уступа (`shimmying`), цели плавно скользят по грани, и IK-решатель реалистично переставляет руки персонажа в реальном времени!

Благодаря этой бесшовной связи физики коллизий и геометрии IK, паркур в Zenith ощущается тактильным и живым.

## 5.5 AST WorldGen: Математический конструктор ландшафта

### В чём проблема?
Обычно генерация мира зашита глубоко в коде: `double density = sampleNoise(x, z) * y - 0.5;`. Если геймдизайнер хочет сделать террасированные горы или сгладить низины, ему нужно идти к программисту, менять код, пересобирать проект и надеяться, что ничего не сломалось.

### Метафора из жизни: Конструктор LEGO из формул
В **Zenith** мы построили генератор мира на основе **Abstract Syntax Tree (AST)** — абстрактного синтаксического дерева. Каждая математическая операция — это отдельный кубик LEGO (`DensityFunction`):
*   `Noise` — генерирует плавный шум.
*   `Add` / `Multiply` — складывает или умножает шумы.
*   `Terrace` — сглаживает высоты ступеньками.
*   `Square` / `Abs` — математические функции для создания острых хребтов.

Все эти кубики соединяются друг с другом прямо в JSON-файлах конфигурации биомов! Движок парсит этот JSON и строит в памяти ветвистое дерево функций.

### Как это устроено в Java: Паттерн Компоновщик (Composite)
Каждая функция реализует один простой интерфейс, благодаря чему их можно вкладывать друг в друга бесконечно:

```java
// Общий интерфейс для всех математических узлов генерации
public interface DensityFunction {
    double compute(DensityContext context);
}

// Узел сложения двух других функций
public record AddFunction(DensityFunction left, DensityFunction right) implements DensityFunction {
    @Override
    public double compute(DensityContext context) {
        // Рекурсивно вычисляем значения левого и правого поддеревьев
        double leftValue = left.compute(context);
        double rightValue = right.compute(context);
        
        // Возвращаем их сумму!
        return leftValue + rightValue;
    }
}
```

## 5.6 Asynchronous Multi-Noise Pipeline & 5D Climate Space: Живой климат без прямых линий

### В чём проблема?
В простых воксельных играх биомы распределяются примитивно: *«Если тепло и сухо — пустыня, если холодно и влажно — тайга»*. В итоге мир выглядит искусственным, а переходы между биомами идут по строгой сетке.

### Метафора из жизни: Пятимерный кодовый замок и пролитая акварель
В **Zenith** мы используем **5D-пространство климата**. Для каждой точки на карте рассчитывается 5 параметров:
1. **Temperature** (Температура: жара/холод).
2. **Humidity** (Влажность: сухость/сырость).
3. **Continentalness** (Континентальность: близость к океану).
4. **Erosion** (Эрозия: плоские равнины/крутые каньоны).
5. **Weirdness** (Странность: определяет уникальные пики или аномалии).

Чтобы границы биомов не выглядели как разметка на парковке, мы применяем **Domain Warping** (искажение координат). Перед тем как спросить у шума: *«Какая тут температура?»*, мы слегка сдвигаем координаты X и Z другим фрактальным шумом. Это похоже на то, как если бы мы нарисовали чёткие границы краской на бумаге, а потом слегка наклонили её и брызнули водой. Границы потекут, создавая естественные извилистые берега рек, рваные опушки лесов и мысы.

### Domain Warping «на пальцах»

```java
public class ClimateGenerator {
    private final NoiseSource warpNoiseX;
    private final NoiseSource warpNoiseZ;

    public Biome getBiomeAt(int blockX, int blockZ) {
        // 1. Слегка искажаем (сдвигаем) мировые координаты фрактальным шумом
        // warpNoiseX.sample выдаёт число от -1.0 до 1.0. Умножая на 50.0, мы сдвигаем границу до 50 блоков
        double warpedX = blockX + warpNoiseX.sample(blockX * 0.005, blockZ * 0.005) * 50.0;
        double warpedZ = blockZ + warpNoiseZ.sample(blockX * 0.005, blockZ * 0.005) * 50.0;

        // 2. Сэмплируем климат по ИСКАЖЕННЫМ координатам
        double temp = sampleTemperature(warpedX, warpedZ);
        double hum = sampleHumidity(warpedX, warpedZ);
        double cont = sampleContinentalness(warpedX, warpedZ);
        double erosion = sampleErosion(warpedX, warpedZ);
        double weird = sampleWeirdness(warpedX, warpedZ);

        // 3. Ищем ближайший биом в 5D-пространстве климата (математика векторов)
        return BiomeRegistry.findClosest(temp, hum, cont, erosion, weird);
    }
}
```

## 6. Готовимся к IT-собеседованию

### Вопрос 1: В чем заключается проблема Gimbal Lock (шарнирного замка) при использовании углов Эйлера?
* **Ответ:** Gimbal Lock возникает, когда при последовательном повороте объекта вокруг локальных осей (Pitch, Yaw, Roll) одна из осей поворачивается на 90 градусов и совпадает с другой осью в пространстве. Объект теряет одну плоскость вращения (степень свободы). Вращения вокруг оставшихся осей начинают дублировать друг друга, из-за чего 3D-модели начинают хаотично дергаться и крутиться. Кватернионы решают эту проблему, так как они задают вращение сразу в 4D-пространстве вокруг одной произвольной оси.

### Вопрос 2: В чем разница между LERP (линейная интерполяция) и SLERP (сферическая интерполяция) при сглаживании вращений?
* **Ответ:** LERP соединяет две точки прямой линией и плавно двигает объект по ней с постоянной скоростью. Это отлично работает для изменения позиций (Translation). 
* SLERP соединяет две точки дугой на сфере вращений и двигает объект по этой дуге с постоянной *угловой* скоростью. SLERP используется для сглаживания поворотов суставов костей скелета, так как он гарантирует абсолютно плавное вращение без уродливых рывков и ускорений в процессе движения.

### Вопрос 3: Почему алгоритм FABRIK IK предпочтительнее классических методов инверсной кинематики на основе матриц Якобиана?
* **Ответ:** 
  1. *Скорость:* Матрица Якобиана требует тяжелых вычислений частных производных и дорогого обращения матриц ($O(N^3)$), что сильно тормозит процессор. FABRIK работает исключительно на простейших векторных сложениях и делениях за линейное время ($O(N)$), работая в 10 раз быстрее.
  2. *Стабильность:* Якобиан страдает от деления на ноль при выпрямлении конечностей, из-за чего руки монстров могут «взрываться» и улетать. FABRIK абсолютно стабилен в любых позах: если цель недосягаема, конечность просто ровно вытягивается в её сторону.
