Добро пожаловать в Академию Zenith! Прежде чем мы начнём разбирать сложную архитектуру игрового движка, работу с памятью или рендеринг миллионов треугольников, нам нужно заложить очень прочный фундамент. 

Эта глава — огромная, подробная и написана специально для тех, кто только начинает свой путь в программировании, или для тех, кто писал простые скрипты, но никогда не задумывался о том, как алгоритмы работают "под капотом". Мы разберем всё: от того, что такое переменная и массив, до сложных игровых алгоритмов вроде поиска пути $A^*$, битовых масок, деревьев (Octree) и связных списков.

Если вы чувствуете себя неуверенно в алгоритмике — налейте кофе, устройтесь поудобнее. Мы разберем всё на жизненных примерах, чтобы вы могли уверенно пройти любое техническое собеседование и понять каждую строчку исходного кода Zenith.

## 1. Как компьютер думает? (Переменные и Память)

В основе любого программирования лежит **Память**. Вы можете представлять оперативную память (RAM) компьютера как гигантскую пустую парковку, расчерченную на миллионы одинаковых парковочных мест (байт). Каждое место имеет свой уникальный номер (адрес в памяти).

Когда вы создаете **переменную** в коде, вы арендуете одно или несколько парковочных мест и ставите туда коробку с данными.

```java
int playerHealth = 100;
String playerName = "Steve";
```

* `int` (Integer, целое число) занимает маленькую коробку (ровно 4 байта). Компьютер точно знает её размер с самого начала.
* `String` (Строка текста) — это длинный составной объект, состоящий из множества символов (букв). Он может занимать десятки парковочных мест и его размер может меняться.

### Типы данных: Примитивы vs Объекты
В языке Java (на котором написан Zenith) все переменные жестко делятся на два лагеря:
1. **Примитивы** (`int`, `float`, `boolean`, `long`, `byte`, `short`, `double`, `char`). Это простые, быстрые числа или логические значения (`true`/`false`). Они хранятся прямо в "рабочем столе" процессора, в специальной сверхбыстрой памяти под названием **Стек (Stack)**. Доступ к ним мгновенный.
2. **Объекты** (`String`, `Zombie`, `Block`, массивы). Это сложные, тяжелые структуры. Они хранятся на гигантском медленном складе — в **Куче (Heap)**. А у вас на столе (в стеке) лежит только маленькая бумажка с адресом этого склада. Эта бумажка называется **Ссылкой (Reference)**. 

> [!IMPORTANT]
> **Золотое правило движков (Zero-Allocation):** Примитивы — это скорость. Создание новых Объектов — это мусор на складе, который потом придется убирать специальному уборщику — Сборщику Мусора (Garbage Collector). Когда уборщик выходит на работу, игра "зависает" (фризит). Подробнее об этом мы поговорим в Главе 1.

## 2. Логика и Ветвления (Как принимать решения)

Программа без логики — это просто продвинутый калькулятор. Чтобы игра реагировала на действия игрока, нужны условия `if / else`.

Представьте охранника (фейс-контроль) в ночном клубе. У него есть строгая инструкция (алгоритм):
* Если гостю меньше 18 лет — не пускать.
* Иначе, если у гостя есть VIP-карта — пустить в VIP-зону.
* Во всех остальных случаях — пустить на общий танцпол.

В коде это выглядит в точности так же:
```java
// Проверяем возраст
if (player.age < 18) {
    player.kickOut();
    
// Слово "else" означает "иначе"
} else if (player.hasVipCard) {
    player.teleportToVip();
    
// Если ни одно условие выше не сработало
} else {
    player.spawnInMainHall();
}
```

Процессор проверяет эти условия строго сверху вниз. Как только одно из условий совпало (вернуло логическое `true`), он выполняет код внутри фигурных скобок `{ }` и **полностью игнорирует** все остальные проверки ниже.

## 3. Массивы и Циклы (Почтовые ящики)

Нам постоянно нужно хранить списки чего-либо: список игроков на сервере, список предметов в инвентаре, список зомби. Для этого используются **Массивы (Arrays)**.

Представьте себе длинный ряд почтовых ящиков на стене подъезда. У каждого ящика есть свой порядковый номер — **Индекс**.
В программировании (из-за исторических причин работы с памятью) нумерация ящиков всегда начинается с **нуля**, а не с единицы. 

```java
// Создаем шкафчик ровно на 3 ячейки
// В Java длина массива фиксирована. Вы не можете положить 4-й предмет!
String[] inventory = new String[3];

// Кладём вещи в ячейки по их индексу (от 0 до 2)
inventory[0] = "Меч";
inventory[1] = "Зелье";
inventory[2] = "Яблоко";
```

### Как прочитать массив? (Циклы)
Если мы хотим вывести в консоль всё, что есть в инвентаре, мы не будем писать `print(inventory[0]); print(inventory[1])...`. Мы используем **Цикл `for`**.

Цикл — это робот-почтальон, которому вы даете три команды: «С какого ящика начать», «До какого ящика идти» и «Какой шаг делать».

```java
// Робот начинает с ящика i = 0.
// Пока номер ящика i меньше длины массива (3), он открывает ящик.
// После каждого шага он делает i++ (переходит к следующему ящику).
for (int i = 0; i < inventory.length; i++) {
    System.out.println("В ячейке " + i + " лежит: " + inventory[i]);
}
```

## 4. Словари и Массивы как тумблеры (O(1))

Массивы удобны, но они скрывают в себе магию невероятной скорости. 
В сложных архитектурах (например, когда мы обрабатываем нажатия тысяч кнопок на клавиатуре) нам нужно мгновенно знать: зажата ли сейчас клавиша «Пробел»?

Представьте, что у нас есть клавиатура. Мы создаем массив на 256 ячеек, где каждая ячейка (от `0` до `255`) — это логический тумблер (состояние `true` или `false`). 
Номер клавиши «Пробел» в системе Windows — `32`. Клавиши «Enter» — `13`.

Когда игрок физически нажимает Пробел, операционная система передает нам сигнал, и мы делаем так:
```java
keys[32] = true; // Клавиша зажата!
```

А когда внутриигровая физика хочет спросить, прыгает ли игрок в данный кадр, она просто проверяет:
```java
if (keys[32] == true) {
    player.jump();
}
```

> [!TIP]
> **Почему это невероятно быстро?** 
> Нам не нужно перебирать весь массив и спрашивать каждую кнопку по очереди: «Эй, ты нажата? А ты?». Мы знаем точный числовой номер Пробела (`32`) и обращаемся напрямую к его «ящику». 
> Компьютер знает, где в оперативной памяти начинается массив `keys`, прибавляет к этому начальному адресу число `32` и мгновенно считывает значение тока. Это работает за долю микросекунды. 
> В компьютерной науке доступ по прямому индексу называется **Доступ за $O(1)$** (константное время). Это Святой Грааль производительности.

## 5. Связные списки (LinkedList) против Массивов (ArrayList)

Массивы прекрасны, но у них есть огромный минус — их длина фиксирована. Если вы создали массив на 3 ячейки, вы не можете положить туда 4-й меч. 
Чтобы решить эту проблему, программисты придумали динамические списки. Самые популярные — `ArrayList` и `LinkedList`. На собеседованиях вас обязательно спросят, в чем между ними разница.

### ArrayList (Динамический массив)
Это обычный массив, но с секретом. Когда он заполняется до краев, он "втихаря" создает новый массив в 1.5 раза больше, копирует туда все старые вещи и удаляет старый массив.
* **Плюс:** Доступ по индексу моментальный ($O(1)$).
* **Минус:** Если вы хотите вставить элемент **в самое начало** (или в середину), компьютеру придется сдвинуть все остальные элементы вправо, чтобы освободить место. Это очень медленно ($O(N)$).

### LinkedList (Связанный список)
Представьте цепочку людей, держащихся за руки. Каждый человек (Узел / Node) держит в одной руке яблоко (данные), а другой рукой крепко держит следующего человека.
Они не стоят по порядку в одной комнате, они могут быть разбросаны по всему городу, но они связаны за руки (ссылками).
* **Плюс:** Если вы хотите вставить нового человека в середину цепи, вам не нужно никого двигать. Вы просто говорите двум людям расцепить руки, и берете между ними нового человека за руки. Вставка в начало или конец мгновенна ($O(1)$).
* **Минус:** Вы не можете крикнуть: "Эй, человек номер 50, дай яблоко!". Вы должны подойти к первому человеку и сказать: "Передай по цепочке до 50-го". Чтобы найти элемент в середине, вам придется пройти по всем людям до него ($O(N)$). В игровых движках `LinkedList` используется редко из-за плохой работы с кэшем процессора.

## 6. Словари / Hash Maps (Картотека в библиотеке)

Списки идеальны, когда у нас есть числовой индекс. Но что, если мы хотим искать данные не по номеру, а по **Имени**? Например, мгновенно узнать здоровье конкретного игрока по его никнейму. Мы не можем написать `health["Alex"]`, если используем массив — массивам нужны числа.

Здесь на помощь приходят **Словари (Map / HashMap)**, также известные как Ассоциативные массивы или Хэш-таблицы.
В словаре данные хранятся парами: **Ключ -> Значение**.

```java
// Создаем пустой словарь (Имя игрока -> Количество ХП)
Map<String, Integer> healthMap = new HashMap<>();

healthMap.put("Alex", 100);
healthMap.put("Steve", 50);

// Как мгновенно узнать здоровье Стива?
int hp = healthMap.get("Steve"); // Вернет 50
```

### Как HashMap работает "под капотом"?
Представьте библиотеку с миллионом книг. Если искать книгу про космос, перебирая каждую полку по очереди, это займет годы. 
Вместо этого в библиотеке есть математическая машина — **Хэш-функция**.

Вы кидаете в эту машину слово `"Steve"`. Машина перемалывает буквы по сложной математической формуле (например, умножая код каждой буквы на простое число 31) и выплевывает число. Например, `42`.
Это число `42` — точный номер полки (индекс скрытого массива внутри `HashMap`). Компьютер мгновенно идет к полке №42 и забирает оттуда цифру `50` (здоровье Стива).

**Коллизии:** Что, если слово `"Alex"` тоже превратится машиной в число `42`? Это называется *Коллизией*. В этом случае на полке №42 будет лежать маленький связный список (`LinkedList`) из двух элементов. Компьютер быстро проверит оба. Но благодаря умной математике коллизии случаются крайне редко.

## 7. Как найти количество уникальных значений? (Set / Множества)

Типичная задача на техническом собеседовании: *«У нас есть огромный список предметов: [Яблоко, Меч, Яблоко, Зелье, Яблоко, Щит...]. Как быстро узнать, сколько тут УНИКАЛЬНЫХ типов предметов?»*

**Плохой алгоритм (Подход новичка):**
Создать новый пустой список. Взять первый предмет, проверить, есть ли он в новом списке. Если нет — положить. Взять второй, перебрать весь новый список, убедиться, что его там нет, положить. 
Если исходных предметов 100 000, компьютер будет постоянно бегать по списку и выполнит **миллиарды проверок** ($O(N^2)$).

**Хороший алгоритм (Подход инженера):**
Использовать **Множество (HashSet)**.
Множество — это специальный "магический мешок", который внутри работает точно так же, как `HashMap`, но хранит только ключи, без значений. Запомните главное правило: **Множество физически не может хранить дубликаты**.

```java
List<String> inventory = Arrays.asList("Яблоко", "Меч", "Яблоко", "Зелье", "Яблоко");

// Создаем пустой "мешок уникальности"
Set<String> uniqueItems = new HashSet<>();

// Ссыпаем весь инвентарь в мешок
for (String item : inventory) {
    // Внутри add() происходит хэширование (превращение слова в число).
    // Если хэш слова "Яблоко" уже занят старым Яблоком, новое просто отбрасывается.
    uniqueItems.add(item); 
}

// Спрашиваем мешок, сколько в нем элементов
int count = uniqueItems.size(); // Вернет 3 ("Яблоко", "Меч", "Зелье")
```
Компьютер прошелся по исходному списку ровно один раз! Время выполнения минимально.

## 8. Алгоритмическая сложность: Big O Notation

Вы наверняка встречали в статьях обозначения вроде $O(N)$ или $O(N^2)$. Эта запись часто пугает новичков, но на самом деле она элементарна.
**Big O** (О большое) — это математический способ измерить, насколько алгоритм **замедляется**, когда данных (обозначаемых как $N$) становится всё больше и больше. 

Мы не измеряем время в секундах, потому что у одного человека старый ноутбук, а у другого мощный сервер. Мы измеряем количество *шагов*, которые нужно сделать процессору.

### 🟢 $O(1)$ — Константное время (Святой Грааль)
Независимо от того, 10 зомби в игре или 1 000 000, алгоритм отработает за одинаковое количество шагов. 
*Пример:* Проверить нажатие кнопки `keys[32] == true` или достать данные из словаря по ключу `healthMap.get("Steve")`. График времени — прямая горизонтальная линия.

### 🟡 $O(N)$ — Линейное время (Нормально)
Если зомби стало в 10 раз больше, алгоритм сделает ровно в 10 раз больше шагов. 
*Пример:* Пробежаться по списку всех зомби одним циклом `for`, чтобы отнять у каждого 1 ХП. Если зомби 100 — будет 100 шагов. Это неизбежный и нормальный процесс.

### 🔵 $O(\log N)$ — Логарифмическое время (Магия бинарного поиска)
Представьте, что вам нужно найти фамилию "Смирнов" в бумажном телефонном справочнике на 1000 страниц. 
Алгоритм $O(N)$ заставил бы вас листать каждую страницу с первой до тысячной.
Алгоритм $O(\log N)$ работает так (при условии, что книга отсортирована по алфавиту):
1. Вы открываете книгу ровно посередине (страница 500). Фамилия начинается на "С", а на 500 странице буква "М". 
2. Вы понимаете, что первая половина справочника (буквы от А до М) вам больше не нужна. Вы отрываете и выбрасываете первые 500 страниц! 
3. Затем открываете середину оставшейся части (от 500 до 1000). 
4. И так далее. На каждом шаге вы отбрасываете ровно половину данных.

Вы найдете человека среди 1000 страниц всего за **10 шагов** ($2^{10} = 1024$). Если страниц будет 1 000 000 — вы найдете человека за **20 шагов**. Если миллиард — за **30 шагов**. Это невероятно, фантастически быстро.

### 🔴 $O(N^2)$ — Квадратичное время (Убийца FPS)
Если зомби стало в 10 раз больше, алгоритм сделает в **100 раз** ($10^2$) больше шагов. Если в 1000 раз больше — в **1 000 000 раз**. 
*Пример:* Двойной вложенный цикл. Если вы захотите проверить столкновение каждого зомби с каждым другим зомби:

```java
// ОПАСНО! Сложность O(N^2)
for (Zombie z1 : allZombies) {
    for (Zombie z2 : allZombies) {
        if (z1 != z2) {
            checkCollision(z1, z2);
        }
    }
}
```
Если на карте 1000 зомби, этот код выполнит **1 000 000 проверок за один единственный кадр**. При 60 кадрах в секунду ваш процессор просто расплавится, и игра покажет 1 кадр в 5 секунд. 

В игровых движках мы избегаем $O(N^2)$ любой ценой. Для решения проблемы столкновений мы делим мир на трехмерную сетку (Spatial Hashing), чтобы зомби проверял коллизии только с 5-10 соседями в своей локальной "клетке", а не со всей картой.

## 9. Алгоритмы Сортировки (Как навести порядок)

Если у вас есть список чисел `[5, 2, 9, 1]`, как заставить компьютер выстроить их по возрастанию `[1, 2, 5, 9]`? Это классическая задача информатики. Существуют десятки способов, но мы рассмотрим два самых ярких.

### Пузырьковая сортировка (Bubble Sort) — $O(N^2)$
Самый простой, но самый медленный алгоритм. 
Представьте пузырек воздуха, который всплывает со дна стакана наверх.
Компьютер сравнивает первые два числа (5 и 2). 5 больше 2? Да. Меняем их местами. Список стал `[2, 5, 9, 1]`.
Сравнивает вторые два (5 и 9). 5 больше 9? Нет. Оставляем.
Сравнивает третьи два (9 и 1). 9 больше 1? Да. Меняем. Список стал `[2, 5, 1, 9]`.
Самое большое число (9) всплыло на самый верх! Теперь алгоритм начинает всё сначала, пока весь список не отсортируется. На больших списках это занимает вечность.

### Быстрая сортировка (QuickSort) — $O(N \log N)$
Это один из самых умных алгоритмов, который встроен под капотом функции `Arrays.sort()` во многих языках.
Он использует подход "Разделяй и властвуй".
1. Компьютер выбирает любое число из списка наугад (называется Опорный элемент или Pivot). Пусть это будет `5`.
2. Он раскидывает остальные числа на две кучки: все, что меньше 5 — налево. Все, что больше 5 — направо.
   Получается: `[2, 1] | 5 | [9]`
3. Теперь число 5 стоит на своем идеальном, финальном месте! 
4. Алгоритм берет левую кучку `[2, 1]` и проделывает с ней то же самое. 

Это происходит в десятки раз быстрее, чем сортировка пузырьком.

## 10. Деревья (Trees) и Графы (Graphs)

Вы привыкли, что данные лежат в массиве (один за другим). Но в играх данные часто ветвятся.

### Что такое Дерево (Tree)?
Представьте семейное древо или структуру папок на вашем компьютере. У вас есть корень (папка `C:\`), в ней лежат ветки (папки `Windows`, `Program Files`), а в них — листья (файлы).
В программировании Дерево — это структура узлов (Node). Каждый узел хранит данные и ссылки на своих "детей".

**Binary Search Tree (Двоичное дерево поиска):**
В таком дереве у каждого родителя не больше двух детей. При этом всё, что меньше родителя, уходит в левую ветку, а всё, что больше — в правую.
Именно эта структура лежит в основе поиска за $O(\log N)$. Когда вы ищете число 42, вы спрашиваете корень (например, 50). 42 меньше 50? Да. Идете налево. Вы мгновенно отсекаете правую половину дерева!

**Octree (Октодерево) в 3D играх:**
В воксельных движках (как Zenith) мир огромен. Чтобы не проверять столкновения везде, 3D-мир делят на огромный куб. Затем этот куб делят на 8 кубиков поменьше (отсюда Oct). Если внутри кубика ничего нет, он остается пустым. Если есть объекты — кубик делят еще на 8. 
Когда летит пуля, мы проверяем пересечение только с гигантским кубом. Если она мимо — мы отсекаем сразу миллионы блоков, не проверяя их по отдельности!

### Что такое Граф (Graph)?
Дерево имеет строгую иерархию (родитель -> ребенок). Но что, если мы моделируем дорожную сеть городов или карту метро? Станция А связана со станцией Б, Б связана с В, а В связана обратно с А. Возникают циклы.
Такая структура "узлов и связей между ними" называется Графом. И для работы с Графами нужны особые алгоритмы, например, алгоритмы поиска пути.

## 11. Магия Битовых Операций (Сдвиги и Маски)

Когда вы будете читать исходный код архитектуры Zenith, вы часто будете видеть пугающие символы: `<<`, `>>`, `&`, `|`. Для разработчиков скриптов (Python, JS) это выглядит как черная магия, но на самом деле это фундамент для самого быстрого в мире манипулирования данными.

### Биты как гирлянда лампочек
Процессор компьютера не понимает десятичные числа (5, 12, 100). Он понимает только электричество: есть ток (`1`) или нет тока (`0`).
Переменная `int` (которая занимает 32 бита) — это просто физическая гирлянда ровно из 32 крошечных лампочек.

Число `5` в двоичной системе выглядит как `...00000101` (горит первая лампочка (1), не горит вторая (2), горит третья (4). Итого 1 + 4 = 5).

### Битовый сдвиг влево `<<` (Умножение на 2)
Операция `5 << 1` означает: взять все горящие лампочки и физически сдвинуть их влево по гирлянде на 1 позицию.
Было: `00000101` (5)
Стало: `00001010` (10)
Сдвиг влево — это самый быстрый аппаратный способ умножить целое число на 2. Процессор делает это напрямую на транзисторах за один такт.

### Побитовое ИЛИ `|` (Включить рубильник)
Представьте, что вы хотите хранить 8 разных настроек игрока (прыгает ли, бежит ли, горит ли в огне, отравлен ли). Новичок создаст 8 разных переменных `boolean`, потратив 8 байт памяти.
Инженер создаст одну переменную типа `byte` (ровно 8 лампочек), потратив всего 1 байт.

Пусть начальное состояние `byte state = 0;` (00000000)
Чтобы сказать "Игрок отравлен" (пусть за это отвечает 3-я лампочка справа), мы берем единицу (`00000001`), сдвигаем её на 2 позиции влево `1 << 2`, получаем `00000100`.
Затем мы применяем операцию **ИЛИ** `|`. Она накладывает две гирлянды друг на друга. Если хотя бы в одной из двух гирлянд на позиции горит лампочка — она будет гореть и в результате.
`state = state | (1 << 2);`
Теперь наш байт `00000100`. Мы включили нужный тумблер!

### Побитовое И `&` (Трафарет / Маска)
А как теперь другой системе игры проверить, отравлен ли игрок? Ей нужна операция **И** `&`.
Она накладывает две гирлянды друг на друга, но лампочка загорается только если она **горит ОДНОВРЕМЕННО в обеих гирляндах**.
Мы делаем своеобразный "картонный трафарет", в котором прорезано только одно отверстие на 3-й позиции (`1 << 2`).
Накладываем этот трафарет на нашего игрока:
`boolean isPoisoned = (state & (1 << 2)) != 0;`
Если результат не ноль — значит, под прорезью в трафарете горела лампочка!

Именно так работает сверхбыстрый безмусорный ввод (`pressedMask |= (1L << keyId)`) из Главы 5.

### Зачем нужны странные маски типа `0x3FFu`?
В Главе 3 вы встретите код упаковки чанков: `(packedData >> 10) & 0x3FFu`.
Это суровый способ запаковать три 10-битные координаты (X, Y, Z) в одно 32-битное число, чтобы видеокарта могла проглотить миллионы вершин и не подавиться.
* `>> 10` — мы сдвигаем гирлянду вправо на 10 позиций, чтобы центральная координата Y встала в самое начало.
* `0x3FF` в шестнадцатеричной системе — это число `1023`, которое в двоичной системе выглядит как `1111111111` (ровно 10 горящих лампочек подряд).
* Накладывая этот "трафарет" через `&`, мы жестко отсекаем (зануляем) весь мусор слева, оставляя только чистую 10-битную координату Y.

## 12. Главные алгоритмы игровых движков (Реальные примеры)

В нашем движке Zenith мы используем несколько классических алгоритмов, без которых не выживает ни одна 3D-игра. Вот они простыми словами, чтобы вы мгновенно узнавали их в исходниках.

### А. Поиск в ширину (BFS - Breadth-First Search)

**Где мы используем:** Расчет глобального освещения от солнца, света от факелов, растекание воды. Это алгоритм обхода Графов и Сеток.

**Как работает на пальцах:**
Представьте каплю, упавшую в лужу — от неё расходятся равномерные круговые волны. 
Алгоритм BFS работает через структуру данных **Очередь (Queue)** (FIFO: кто первый пришел, тот первый ушел).

1. Сначала мы берем стартовый блок (факел с яркостью 15) и кладем его координаты в очередь.
2. Мы достаем первый элемент из очереди. Смотрим на его соседей слева, справа, сверху, снизу, спереди, сзади.
3. Если сосед еще не освещен и там воздух — мы передаем ему яркость `текущая - 1` (то есть 14) и **кладем координаты этого соседа в конец очереди**.
4. Повторяем процесс в цикле `while (!queue.isEmpty())`, пока очередь не опустеет.

Свет плавно растекается по пещере, словно круги по воде, слой за слоем, пока яркость не упадет до 0. BFS гарантирует, что свет найдет самый короткий путь за угол пещеры, не зациклившись.

### Б. Умный поиск пути (Алгоритм $A^*$)

BFS хорош для света, потому что свет светит во все стороны равномерно. Но если умный зомби должен дойти до игрока в сложном лабиринте, пускать круги по всей карте — слишком дорого. Нам нужен направленный алгоритм поиска пути — **$A^*$ (A-Star)**.

**Как работает на пальцах:**
Это тот же BFS, но зомби надевает очки с "эвристикой" (предвидением). Вместо того чтобы проверять все блоки вокруг себя равномерно, он отдает строгий приоритет тем блокам, которые **геометрически ближе к цели** (по прямой линии, игнорируя стены).
Для каждой соседней клетки алгоритм считает два числа:
1. `G` — сколько шагов по лабиринту я уже прошел от старта до этой клетки.
2. `H` (Heuristic) — примерное расстояние от этой клетки до цели "по воздуху" (Манхэттенское расстояние).
Он складывает их `F = G + H` и всегда выбирает идти в клетку с наименьшим общим `F`. Это позволяет зомби целенаправленно и агрессивно бежать в сторону игрока, огибая препятствия, а не исследовать всю карту вслепую.

### В. Трассировка луча (Raycasting / Алгоритм DDA)

**Где мы используем:** Когда игрок кликает мышкой по экрану, чтобы сломать блок или ударить мечом. Экран у нас плоский (2D), а мир объемный (3D). Как движку понять, на какой именно кубик смотрит перекрестие прицела?

**Как работает на пальцах:**
Компьютер пускает математический 3D-вектор (невидимый "лазерный луч") из камеры строго по направлению взгляда. 
Алгоритм **DDA (Digital Differential Analyzer)** позволяет лучу лететь вперед не крошечными миллиметровыми шажками (это требует тысяч проверок и убивает CPU), а четкими прыжками от стенки одного воксельного кубика к стенке следующего. 
На каждом математическом пересечении целочисленной сетки координат луч спрашивает движок: "В вокселе `[x][y][z]` есть твердый блок?". 
* Если там воздух — луч летит к следующей стенке. 
* Если натыкается на камень — Бинго! Мы нашли цель. Расчет прекращается и мы возвращаем координаты камня.

### Г. Отсечение невидимого (Frustum Culling)

**Где мы используем:** Видеокарта мощная, но если отправить ей на отрисовку весь воксельный мир на 32 чанка вокруг (это миллионы полигонов), она умрет. Нам нужен способ не рисовать то, что игрок не видит.

**Как работает на пальцах:**
Ваши глаза (камера в игре) видят только то, что находится спереди. Зона видимости имеет математическую форму усеченной пирамиды (Frustum). У этой пирамиды есть 6 бесконечных плоскостей (левая, правая, верхняя, нижняя, ближняя, дальняя).
Мир состоит из чанков (коробок 16x16x16 блоков, или Bounding Box).
Математика (вычисление Dot Product) позволяет за пару умножений проверить: находится ли коробка чанка **в положительном полупространстве** от всех 6 плоскостей пирамиды?
Если чанк оказался за спиной, сбоку или слишком высоко (вне пирамиды) — алгоритм жестко отсекает его (`Cull`) и даже не отправляет геометрию на видеокарту. Это экономит до 80% производительности GPU.

## 13. Архитектурные паттерны (ООП против ECS)

В классическом обучении программированию вас наверняка учили **Объектно-Ориентированному Программированию (ООП)** и Наследованию.
Вам говорили: "Создай абстрактный класс `Animal`. От него унаследуй `Dog` и `Cat`. У `Cat` добавь метод `meow()`".
Это отлично работает для банковских и бизнес-приложений. Но в игровых движках глубокая иерархия наследования (когда класс `Zombie` наследует `Monster`, который наследует `Entity`, который наследует `GameObject`) — это архитектурный ад. Что если вы захотите создать Ядовитое Дерево, которое не двигается (как дерево), но отравляет (как монстр)? В ООП вы запутаетесь в наследовании.

В современных движках (Unity DOTS, Unreal Mass, и в Zenith) используется паттерн **ECS (Entity-Component-System)** (или строгий Data-Driven подход).

Вместо того чтобы создавать сложные объекты с методами, мы делаем так:
* **Сущность (Entity)** — это просто пустая бирка с числовым ID (например, Сущность №42).
* **Компоненты (Components)** — это плоские мешки с данными, которые мы вешаем на эту бирку. (Мешок `HealthComponent(100)`, мешок `RenderComponent(Модель)`, мешок `AIComponent`). Никакой логики внутри компонентов нет! Только данные.
* **Системы (Systems)** — это функции-рабочие, которые бегают по миру каждый кадр. Рабочий `PhysicsSystem` берет ВСЕ сущности, у которых есть компонент `Position` и `Velocity`, и двигает их. Ему абсолютно плевать, кто это — зомби, летящая стрела или падающий блок песка. Если у тебя есть координаты и скорость — ты будешь двигаться.

Это позволяет геймдизайнерам легко собирать новых монстров "на лету" прямо из конфигурационных JSON файлов, как из конструктора Lego, не написав ни строчки нового Java-кода. Вы просто вешаете на пустую сущность нужные мешки-компоненты. Подробнее мы разберем великолепие Data-Driven дизайна в Главе 9.

## 14. Функции и Методы (Роботы-помощники)

Если вы пишете весь код в одном месте (в `main`), ваша программа быстро превратится в нечитаемую "простыню" на тысячи строк. Чтобы этого избежать, код разбивают на **Функции** (в ООП они называются **Методами**).

Функция — это как отдельный маленький робот на заводе. Вы даете ему материалы (параметры), он делает свою узкую задачу и возвращает вам готовый продукт (результат).

```java
// Робот, который умеет только складывать два числа
public int addNumbers(int a, int b) {
    int sum = a + b;
    return sum; // Возвращаем результат начальнику
}

// Начальник вызывает робота:
int result = addNumbers(5, 10); // result станет 15
```
**Зачем это нужно?** 
1. **Переиспользование:** Вам не нужно 10 раз писать формулу расчета урона. Вы пишете функцию `calculateDamage()` один раз и вызываете её откуда угодно.
2. **Чистота:** Код вида `if (isPlayerDead())` читается в сто раз легче, чем `if (player.health <= 0 && player.isSpawned && !player.isInvincible)`.

## 15. Рекурсия (Матрешка или День Сурка)

**Рекурсия** — это когда функция вызывает саму себя внутри себя. Звучит как бесконечная петля или фильм "Начало", но это невероятно мощный инструмент, особенно при работе с Деревьями и Графами.

Классический пример — вычисление Факториала (например, $5! = 5 \times 4 \times 3 \times 2 \times 1$).

```java
public int factorial(int n) {
    // Базовый случай (УСЛОВИЕ ВЫХОДА ИЗ МАТРЕШКИ)
    if (n == 1) {
        return 1;
    }
    // Функция вызывает САМУ СЕБЯ с другим параметром!
    return n * factorial(n - 1);
}
```

**Как это работает на пальцах:**
Вы просите робота посчитать `factorial(3)`.
Робот говорит: "Я не знаю, сколько это. Но это $3 \times factorial(2)$". Он ставит свою работу на паузу и создает клона, чтобы тот посчитал `factorial(2)`.
Клон говорит: "Я не знаю. Но это $2 \times factorial(1)$". Он ставит работу на паузу и создает второго клона для `factorial(1)`.
Второй клон видит Условие Выхода (`if n == 1`). Он радостно возвращает `1`.
Первый клон умножает $2 \times 1 = 2$ и возвращает `2`.
Главный робот умножает $3 \times 2 = 6$ и возвращает финальный ответ. Матрешка схлопнулась!

> [!WARNING]
> **Опасность рекурсии:** Если вы забудете написать условие выхода (Базовый случай), клоны будут создаваться бесконечно. Оперативная память (Стек), куда складываются "задания на паузе", переполнится, и игра вылетит с ошибкой `StackOverflowError`.

## 16. Интерфейсы и Полиморфизм (Контракты)

Когда вы проектируете игру, у вас может быть 50 разных видов оружия (Меч, Лук, Магический посох, Бомба). Если вы попытаетесь написать обработчик атаки для каждого отдельно, вы сойдете с ума:

```java
// Ужасный код новичка
if (weapon is Sword) {
    ((Sword)weapon).slash();
} else if (weapon is Bow) {
    ((Bow)weapon).shootArrow();
} else if (weapon is Staff) {
    ((Staff)weapon).castSpell();
}
```

Здесь на помощь приходят **Интерфейсы**. Интерфейс — это строгий бумажный Контракт.
Вы создаете интерфейс `IWeapon`, в котором написано: *"Каждый, кто подпишет этот контракт, ОБЯЗАН иметь метод `attack()`"*.

```java
public interface IWeapon {
    void attack();
}

// Меч подписывает контракт (implements)
public class Sword implements IWeapon {
    public void attack() {
        System.out.println("Взмах мечом!");
    }
}
// Лук подписывает контракт
public class Bow implements IWeapon {
    public void attack() {
        System.out.println("Выстрел стрелой!");
    }
}
```

Теперь ваш код атаки в игре выглядит идеально и занимает одну строчку. Это называется **Полиморфизм**:
```java
// Нам плевать, что конкретно в руках у игрока. 
// Раз это IWeapon, значит у него точно есть метод attack().
IWeapon currentWeapon = player.getEquippedWeapon();
currentWeapon.attack(); 
```
Если вы завтра добавите в игру "Лазерную пушку", вам не придется переписывать старый код. Пушка просто подпишет контракт `IWeapon` и сама разберется, как стрелять лазерами.

## 17. Паттерн Singleton (Одиночка)

Паттерны проектирования — это готовые чертежи для решения частых проблем. Вы будете часто встречать их в коде Zenith. Самый известный (и часто критикуемый) паттерн — Одиночка (Singleton).

Представьте, что у вас есть менеджер звуков `SoundManager`. Вы хотите, чтобы во всей игре существовал **только один** менеджер звуков. Если их будет два, они начнут перебивать друг друга, и музыка будет играть дважды.

Чтобы никто не смог случайно создать второго менеджера через `new SoundManager()`, мы делаем конструктор приватным (запрещаем создавать объекты снаружи), а сам объект храним внутри самого класса.

```java
public class SoundManager {
    // 1. Храним единственный экземпляр внутри класса
    private static SoundManager instance;

    // 2. Делаем создание закрытым для чужих!
    private SoundManager() {
        System.out.println("Аудио движок запущен");
    }

    // 3. Отдаем инстанс всем желающим
    public static SoundManager getInstance() {
        if (instance == null) {
            instance = new SoundManager(); // Создаем только при первом запросе
        }
        return instance;
    }
    
    public void playExplosion() { ... }
}
```
Теперь из любой точки игры (даже из самого глубокого скрипта) вы можете написать:
`SoundManager.getInstance().playExplosion();`
Это глобальная точка доступа. Это удобно, но злоупотреблять Синглтонами нельзя — они делают код запутанным и сложным для тестирования.

## 18. Паттерн Фабрика (Factory)

Что, если вам нужно постоянно создавать разных врагов в зависимости от биома (уровня)?
Вместо того чтобы размазывать `new Zombie()` и `new Skeleton()` по всему коду, вы создаете завод (Фабрику), который берет эту грязную работу на себя.

```java
public class EnemyFactory {
    public static Enemy spawnEnemy(String biome) {
        if (biome.equals("Forest")) {
            return new Spider();
        } else if (biome.equals("Graveyard")) {
            return new Zombie();
        } else if (biome.equals("Hell")) {
            return new Demon();
        }
        return new Slime(); // По умолчанию
    }
}
```
Теперь, когда генератор уровней хочет заспавнить монстра, он не думает о том, какие классы существуют. Он просто делает заказ на завод:
`Enemy mob = EnemyFactory.spawnEnemy("Hell");`
Если вы добавите нового крутого босса, вы измените код только в одном месте — внутри Фабрики. Это принцип инкапсуляции и единой ответственности.
